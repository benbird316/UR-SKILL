# 反模式

> 加载阶段：步骤 4（校验）
> 技术翻译领域的常见反模式与规避策略。涵盖传统翻译反模式（1-7）和 LLM Agent 翻译特有反模式（8-10）。

## 1. 通用反模式引用

以下通用反模式由 UR-SKILL 体系统一定义，本 SKILL 不再重复罗列：
- 占位符残留
- 规范越位
- 示例污染
- 架构混淆
- 检查缺失
- 盲区逃避
- 风险边界滥用
- 切面填充
- 盲区甩锅
- 能力降级
- 全局规则步名不一致

完整定义见 UR-SKILL 通用反模式库。以下仅列出**技术翻译领域特有**的反模式。

## 2. 领域特有反模式

### AP-01 猜测术语

**表现**：遇到不确定的技术术语时不联网查证，直接基于字面意思或经验猜测英译。

**为什么出现**：想提高效率，觉得「这个术语大概就是 XXX 吧」，不值得花一轮搜索；或者搜索配额有限，想省着用。

**危害**：
- 一旦术语译错，全文一致性框架崩溃
- 读者对整篇文档的专业性产生质疑
- 错误术语可能被下游系统引用，造成术语体系污染

**检测方法**：检查术语映射表中每条译法是否有来源记录；无来源且未标注 Inferred 的即为猜测。

**规避策略**：
- 按步骤 2 执行联网查证，搜索格式：`"{中文术语}" English technical term`
- 优先从官方文档确认
- 确实无法查证的 → 标注 `[Inferred: confidence low]`，明确告知是推断

---

### AP-02 多样性译法

**表现**：同一中文术语在文档不同位置被译为不同的英文词（如「配置」时而译为 `configuration`，时而译为 `settings`），译者以为「换词显得词汇丰富」。

**为什么出现**：受文学翻译思维影响，觉得重复用同一个词显得词汇量不够，想追求表达多样性。

**危害**：
- 在技术文档中，读者会认为 `configuration` 和 `settings` 是两个不同的概念，造成理解混乱
- 违反 ISO 5060:2024 Terminology 错误类别，属于 Major 级错误
- 术语映射表失去作用

**检测方法**：扫描全文高频术语（出现 ≥3 次）的每一次出现，对比译法是否统一。

**规避策略**：
- 建立术语映射表后全文严格一致
- 技术翻译追求一致性，不是文学翻译追求的多样性
- 确实需要不同译法的（上下文不同概念）→ 在翻译说明中标注 context-dependent

---

### AP-03 逐字翻译中文套话

**表现**：将中文技术文档中的冗余套话逐字译为英文，如：
- 「进行配置」→ `carry out the configuration`（应为 `configure`）
- 「实现功能」→ `implement the functionality`（应为 `supports`）
- 「需要注意的是」→ `it is necessary to note that`（应为 `Note:` 或省略）

**为什么出现**：逐字翻译最省力，不需要思考中文表达在英文中的对应习惯；或者对英文技术写作风格不熟悉，直译最安全。

**危害**：
- 译文冗长、读起来像中式英语（Chinglish）
- 降低技术文档的可读性和专业感
- 违反 Microsoft/Google 技术写作风格指南

**检测方法**：扫描译文中是否高频出现 `carry out`、`implement the`、`it is necessary to`、`relevant`、`corresponding` 等中式英语特征词。

**规避策略**：
- 识别中文冗余套话，翻译时精简为符合英文技术写作习惯的表达
- 中文中「进行」「实现」「相关」「相应」等词在 90% 的情况下没有独立语义，英译时直接省略
- 参照 fix-patterns.md 中的冗余套话转换矩阵

---

### AP-04 翻译代码/标识符

**表现**：翻译技术文档时误入代码块或行内代码，改动其中的内容：
- 翻译代码注释中的函数名
- 将行内代码中的字符串翻译为英文
- 改动占位符变量的格式

**为什么出现**：模型注意力扩散，看到中文就想翻译，没有严格区分代码区和正文区；或者觉得「代码注释也是中文，也该翻译」。

**危害**：
- 代码不可运行，变量引用断裂
- 技术准确性丧失
- 用户复制粘贴代码直接报错

**检测方法**：对比原文和译文的代码块内容，检查标识符、变量名、函数名是否被改动。

**规避策略**：
- 代码块内容完全不改动（包括代码注释中的标识符）
- 行内代码原样保留
- 占位符变量（`{xxx}`、`%s`、`<xxx>`）原样保留
- 代码注释中的中文说明是否翻译 → 视用户需求而定，默认不译

---

### AP-05 添加解释

**表现**：译者觉得「这个概念翻译成英文后读者可能不懂」，在译文中添加原文没有的解释或背景说明。

**为什么出现**：好心，想让读者更容易理解；或者觉得「只翻译不解释显得不够专业」。

**危害**：
- 超出翻译职责，引入原文没有的信息
- 如果解释有误，后果更严重
- 违反「翻译只做语言转换」的基本原则

**检测方法**：逐句对照原文和译文，检查译文中是否有原文找不到对应内容的句子。

**规避策略**：
- 翻译只做语言转换，不做知识科普
- 确有必要标注复杂度特别高的概念 → 使用 `[Translator's note: ...]` 格式，但应极为克制
- 大量解释需求 → 属于技术写作任务，不是翻译任务

---

### AP-06 中文标点英文文本

**表现**：英文译文中出现中文标点（全角逗号、中文引号「」、中文书名号《》）。

**为什么出现**：输入法切换不及时，或者复制粘贴时带过来的；模型输出时偶尔会混入源语言标点。

**危害**：
- 极不专业，英文读者无法正确解析标点
- 影响排版和可读性
- 属于 ISO 5060:2024 Linguistic conventions 错误类别

**检测方法**：正则扫描译文中的全角字符（`[\uff00-\uffef]`）和中文标点。

**规避策略**：
- 英译文本全部使用英文标点（半角）
- 书名号 → 斜体或引号
- 步骤 4 校验时执行全角字符扫描

---

### AP-07 忽略目标读者差异

**表现**：不管文档的受众是谁，用同一套翻译策略——全是长句全是术语全是被动语态。

**为什么出现**：一套策略走天下最省事，不需要判断读者类型；或者觉得「技术文档就该是正式风格」。

**危害**：
- 面向开发者的文档写得像学术论文，增加阅读负担
- 面向普通用户的产品说明写得像技术规范，用户看不懂
- 风格与目标读者不匹配，文档效果大打折扣

**检测方法**：检查步骤 1 是否有目标读者判定；译文风格是否与判定的读者类型匹配。

**规避策略**：
- 步骤 1 必须判定目标读者（开发者/最终用户/学术审稿人）
- 步骤 3 翻译时采用对应的风格策略
- API 文档 → 祈使句、简洁
- 学术论文 → 正式、被动语态适度使用
- README → 友好但不随意

### AP-08 术语幻觉【LLM Agent 特有】

**表现**：LLM 翻译时虚构看似专业但无法在权威源验证的术语译法。例如将「数据脱敏」译为 `data anonymization masking protocol`（看起来像标准术语，但实际上 ISO/IEC 标准中使用的是 `data masking` 或 `data anonymization`，不存在合并的标准化术语）。

**为什么出现**：LLM 倾向于输出「看起来合理」的内容，遇到不确定的术语时会基于训练数据中的模式「补全」一个看似专业的译法；模型本身不知道自己不知道。

**危害**：
- 虚构术语一旦被采用，可能被下游系统作为标准术语引用，造成术语体系的系统性污染
- 根据 HalloMTBench 研究（Wu et al., 2025），17 个主流 LLM 的翻译幻觉率为 33%-60%
- 属于 Source Detachment（源脱离）中 Addition 子类型的高发形态

**检测方法**：
- 每个术语译法必须至少有 1 个权威来源验证
- 对声称来自 ISO/IEEE 标准的术语，必须在官方文档中定位到具体出处
- 对复合术语（两个标准术语合并），拆分为独立术语分别验证

**规避策略**：
1. 步骤 2 联网查证时，搜索术语在权威源的精确出现形式
2. 如果候选译法在权威源中不存在，标记为 Low 置信度
3. 在术语表中标注：若无权威源确认 → `[Unverified: no authoritative source found]` → 回归到已验证的译法

---

### AP-09 风格漂移【LLM Agent 特有】

**表现**：长文档翻译中，前半段（如前 500 词）使用正式学术风格，后半段（如后 500 词）退化为口语化、冗长、或中式英语风格。

**为什么出现**：LLM 的注意力随上下文长度衰减，长文档后半段对初始风格指令的遵循度下降；Lost in the Middle 效应导致中间和末尾的风格约束被稀释。

**危害**：
- 风格不统一使专业文档丧失可信度
- 读者产生「这部分是另一个人写的」或「这部分是机器翻译的」印象
- 属于 ISO 5060:2024 Style 错误类别

**检测方法**：
- 分段采样：取前 1/3、中 1/3、后 1/3 各至少 1 段
- 对比指标：平均句长、被动语态比例、术语密度、语域标记词
- 如果首尾段在这些指标上偏差超过 30%，则存在风格漂移

**规避策略**：
1. 步骤 3 翻译时，先翻译全文后再统一做一次风格审查
2. 步骤 4 校验时必须执行均匀采样（前 1/3、中 1/3、后 1/3 各至少 1 段）
3. 建立文档级风格锚点（首段作为风格基准），后续段落对齐此基准
4. 对长文档（>2000 字），使用「参数锚定」策略：每翻译一段前，回顾首段的句式风格

---

### AP-10 长文档一致性衰减【LLM Agent 特有】

**表现**：同一术语在长文档的不同位置出现不同译法；语气在正式/半正式间来回切换。例如：
- 前 1/3：「容器编排」→ `container orchestration`
- 中 1/3：「容器编排」→ `container scheduling`（混淆了编排和调度）
- 后 1/3：「容器编排」→ `container management`（退化为泛化术语）

**为什么出现**：长文档中，术语映射表的上下文逐渐被稀释，模型在后续段落中「忘记」了之前确定的译法，重新生成新的译法。

**危害**：
- 根据 ISO 5060:2024 的 Terminology 错误类别，术语不一致被归类为可导致译文 Fail 的 Major/Critical 错误
- HalloMTBench 研究确认长文本（>499 字符）是 LLM 翻译幻觉的主要触发因素之一

**检测方法**：
- 全文扫描高频术语（出现 ≥3 次）的每一次出现，确认译法一致
- 逐段检查语域标记词，确认无突变
- 特别注意段落边界处的术语使用——这是「衰减」最常发生的区域

**规避策略**：
1. 步骤 2 建立术语映射表后，步骤 3 翻译时严格执行「一个术语 → 一个译法」
2. 步骤 4 校验时，扫描每个高频术语的所有出现位置，确认译法统一
3. 对跨段落术语，建立全局术语 → 译法映射表，执行全文一致性扫描
4. 在术语映射表的 Notes 列中，对 context-dependent 术语单独标注上下文 → 译法映射
