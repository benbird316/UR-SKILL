# 故障诊断

> 加载阶段：步骤 4（校验）
> 技术翻译执行过程中的常见故障及修复方案。涵盖一般翻译问题（T01-T09）和 AI 翻译特有故障（T10-T12）。

---

## 一般翻译故障

### T01 搜索不到术语的标准英译

**症状**：联网搜索多次后仍找不到权威来源的对应英文术语。

**根因**：
- 术语是中国特有技术概念，英文生态中无直接对应
- 术语属于新兴领域，尚未形成标准译法
- 搜索关键词选择不当，未命中正确领域

**修复动作**：
1. 尝试搜索该术语的技术定义而非翻译：`what is {术语概念} in {技术领域}`
2. 如果是中国特有技术概念（如「信创」「等保」），使用音译 + 解释策略：
   - 格式：`{拼音} ({英文解释})`，如 `Xinchuang (China's IT Application Innovation initiative)`
3. 如果属于新兴技术术语，基于构词规律推断译法，标注 `[Inferred: confidence low]`
4. 在术语映射表 Notes 列记录无法查证的原因

---

### T02 同一术语不同上下文译法不同

**症状**：术语映射表建立后，发现同一术语在不同上下文中对应的英文概念不同。

**根因**：中文术语存在多义性，不同技术领域指向不同英文概念（如「端口」在网络=port，在硬件=connector）。

**修复动作**：
1. 确认不同上下文是否确实对应不同概念，而非仅仅是表达习惯差异
2. 确认是不同概念 → 分别建立映射，在术语表中标注 `context-dependent`
3. 只是语气或表达习惯差异 → 保持统一，选择最通用的译法
4. 在翻译说明中列出上下文 → 译法的对应关系

---

### T03 长句拆分后英文显得破碎

**症状**：按短句优先原则拆分中文长句后，英文段落由多个短句组成，读起来不流畅、缺乏逻辑衔接。

**根因**：拆分过度，只关注了句长而忽略了句间逻辑关系；中文靠意合连接，英文靠形合连接。

**修复动作**：
1. 使用连接词（however, therefore, additionally）建立句子之间的逻辑关系
2. 如果多个短句表达同一逻辑链，考虑用分号或破折号合并
3. 平衡原则：目标不是「每句都短」，而是「每句都可被清晰理解」
4. 推荐句长 15-25 词，允许适度浮动

---

### T04 代码块中的中文注释

**症状**：代码块内包含中文注释，默认规则禁止翻译代码块内容。

**根因**：代码块属于非翻译区，但其中的注释是自然语言，用户可能期望翻译。

**修复动作**：
- 默认策略：代码块内中文注释不翻译（保持原样）
- 如果用户明确要求翻译代码注释 → 在步骤 1 确认需求，单独提取注释翻译后回填
- 在翻译说明中标注：「代码注释已翻译」或「代码注释保持原文」

---

### T05 Markdown 表格翻译后列宽失衡

**症状**：中文术语较短但英文译法很长，导致 Markdown 表格源码排版混乱、列不对齐。

**根因**：中英文本长度差异大（中文 2-3 字的术语英文可能有 20+ 字符），Markdown 表格的源码对齐依赖字符宽度。

**修复动作**：
1. 对表格列宽不做强制对齐要求（Markdown 渲染器会自动处理显示效果）
2. 如果英文术语过长（>30 字符），首次出现时标注缩写，后续表格中使用缩写
3. 优先保证内容准确，源码排版美观度次之

---

### T06 原文包含技术错误

**症状**：翻译过程中发现原文本身包含技术错误（错误的数据、矛盾的说法、过时的信息）。

**根因**：原文作者的技术错误，不属于翻译职责范围。

**修复动作**：
1. 不修改原文的技术内容（翻译只做语言转换）
2. 在翻译说明中标注：`[Original text appears to contain a technical inaccuracy: ...]`
3. 建议用户在翻译完成后审校技术内容的正确性
4. 禁止自行「纠正」原文的技术错误

---

### T07 非 Markdown 格式输入

**症状**：用户提供的是纯文本、PDF 提取文本或其他非标准格式，缺少 Markdown 结构标记。

**根因**：输入格式多样，SKILL 默认针对 Markdown 优化，非 Markdown 文档的格式保持策略需要调整。

**修复动作**：
1. 在步骤 1 明确标注文档格式
2. 对非 Markdown 格式，格式保持的侧重点从「Markdown 语法」调整为「段落结构 + 列表 + 编号」
3. 在翻译说明中标注格式限制和处理策略
4. 如果用户期望输出 Markdown 格式 → 在翻译时补充合理的 Markdown 结构

---

### T08 翻译中发现大量未识别术语

**症状**：翻译到后半部分才发现新的高频术语，未在步骤 1 的术语清单中。

**根因**：步骤 1 的术语提取基于前半段文档，长文档后半段出现新领域术语；或某些术语在前期出现频率低，未达到提取阈值。

**修复动作**：
1. 不中断翻译回到步骤 2，继续使用推断译法完成初译
2. 在步骤 3 中标记新发现的术语
3. 在步骤 4（校验）中对新术语补充一次快速联网查证（1 轮搜索）
4. 更新术语映射表，补充新术语的译法和置信度

---

### T09 目标读者专业背景不确定

**症状**：步骤 1 无法确定目标读者的技术背景深度，影响风格和术语策略的选择。

**根因**：用户未明确说明目标受众，文档本身也没有足够线索判断。

**修复动作**：
1. 默认假设：目标读者具备该技术领域的基础知识，但非领域专家
2. 术语策略：首次出现的专业术语标注全称，后续使用缩写
3. 风格策略：采用标准技术文档风格，不过度简化也不过度学术化
4. 在翻译说明中标注：`[Target audience assumed: technical professionals with basic domain knowledge]`
5. 建议用户根据实际受众做最终适应性调整

## AI 翻译特有故障

### T10 语言混用 / 指令脱离

**症状**：英文译文中混入中文字符或整句中文未翻译。例如译文前半段为纯英文，中间突然出现一段未被翻译的中文。

**根因**：
- LLM 「忘记」了翻译指令，属于 Instruction Detachment（指令脱离）型幻觉
- RL 微调的模型更容易触发此问题（Alibaba SASFT, 2026）
- 长文档中后段注意力衰减导致指令遵循度下降

**修复动作**：
1. 重新翻译混入的中文段落
2. 如果反复出现，降低单次翻译的文本量——将长文档拆分为每段不超过 500 词的段落分别翻译
3. 对 RL 模型，在步骤 5 对抗测试中额外执行「纯英文输出」检查
4. 步骤 4 校验时，对译文全文执行中文字符扫描（正则：`[\u4e00-\u9fff]`）

---

### T11 源内容脱离 — 增译 / 漏译

**症状**：
- **增译（Addition）**：英文译文添加了原文没有的解释性内容
- **漏译（Omission）**：英文译文缺少原文中的关键信息

**根因**：
- LLM 脱离源文本自行发挥，属于 Source Detachment（源脱离）型幻觉
- 信息密度低的段落（短句、列表项、表格单元格）是高发区域
- 不同模型倾向不同：部分模型增译倾向强，部分模型漏译倾向强

**修复动作**：
1. 删除无来源的新增内容——翻译只做语言转换，不做知识科普
2. 补充被遗漏的信息，确保语义完整性
3. 如果某段落反复出现增译/漏译，对该段执行回译验证（英译中后对比原文）
4. 步骤 4 对表格、列表、标题等短文本区域做逐项核对

---

### T12 术语上下文歧义

**症状**：同一中文术语在不同段落指向不同英文概念，但建立术语映射表时未区分上下文，导致全文统一使用同一译法，部分段落术语错误。

**根因**：
- 中文术语的多义性在步骤 1 未被充分识别
- LLM 倾向于「一个术语一个译法」的简化处理，忽略了上下文差异
- 长文档中不同章节可能属于不同子领域

**修复动作**：
1. 对多义术语建立上下文 → 译法映射表，而非单一译法
2. 在术语映射表的 Notes 列中标注 `context-dependent`，并列出各上下文对应的译法
3. 步骤 4 执行逐段术语语境检查（非仅全文一致性扫描）
4. 步骤 2 查证时，对多义术语搜索 `{术语} {领域A} vs {领域B}` 确认译法差异
