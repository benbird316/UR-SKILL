# Scripts Design Guide

> Only teaches how to write executable scripts under scripts/. Scripts are generated by the LLM and may be executed by an automation system.
> To determine whether a script needs to be generated, see skill-package-design-guide.md §2.

---

## §1 Safety Specifications

### 1.1 Five Safety Principles

| Principle | Description |
|:---|:---|
| **Read-Only First** | Scripts prioritize reading, do not modify the file system |
| **Idempotence** | Multiple executions produce consistent results |
| **Self-Contained** | Does not depend on external environments |
| **Least Privilege** | Only requests the necessary permissions |
| **Input Validation** | Validates all inputs, rejects invalid input |

### 1.2 Prohibited Operations

**MUST NOT** perform file system modifications (delete, write, rename).
**MUST NOT** perform network requests (HTTP/HTTPS, API calls, downloads).
**MUST NOT** execute system commands (shell commands, subprocesses, system calls).
**MUST NOT** modify environment variables (PATH, system configuration).
**MUST NOT** access sensitive information (passwords, keys, tokens, configuration files).

### 1.3 Permitted Operations

- Read files (read-only access, no modification)
- Regex matching (text scanning, pattern recognition)
- String processing (splitting, concatenation, formatting)
- Mathematical calculations (numeric operations, statistics)
- Data structure operations (list, dict, set operations)

## §3 Script Types

| Type | Purpose | Safety Level | Example |
|:---|:---|:---:|:---|
| **Scan Script** | Static checks, identify issues | Safe | Anti-pattern scan, placeholder detection |
| **Validation Script** | Validate format, constraints, compliance | Safe | YAML frontmatter validation |
| **Generation Script** | Generate content, fill templates | Medium Risk | Batch generate SKILL files |
| **Conversion Script** | Format conversion, data migration | Medium Risk | Markdown to YAML |
| **Deployment Script** | Deploy to production environment | High Risk | Auto-deployment, CI/CD pipeline |

> Medium risk and above scripts require human review and cannot be executed automatically.

## §4 Regex Writing Rules in YAML Configuration Files

When scripts load regular expressions via YAML, the following rules must be followed to avoid escaping errors and runtime `re.PatternError`.

| Scenario | Rule | Example (target regex: `\[`) | Anti-example (causes double backslash) |
|:---|:---|:---|:---|
| Single-quoted string | Use `\\` to represent one backslash | `'\\['` | `'\\\\['` |
| Double-quoted string | Use `\\` to represent one backslash | `"\\["` | `"\\\"` |
| Validation method | Call `re.compile(pattern)` immediately after loading | `re.compile(config['pat'])` | Only compiled at runtime via `findall` |
| Readability first | Complex regex should be placed in script constants (raw string), not in YAML | `r"^\d+$"` | Writing `'^\\d+$'` in YAML |

## §5 Checklist

- [ ] Script has safety level annotation (safe/medium-risk/high-risk)
- [ ] Script has side effect description (none/read/write/network)
- [ ] Script has input validation function
- [ ] Script does not perform high-risk operations (delete, write, network, system commands)
- [ ] Script outputs to stdout, does not write directly to files
- [ ] Script has error handling (try/except)
- [ ] Script has logging (operation log for audit purposes)
- [ ] Medium risk and above scripts have undergone human review
- [ ] Script is self-contained, does not depend on external environments
- [ ] Script is idempotent, multiple executions produce consistent results
- [ ] File < 200 lines; split if exceeded
