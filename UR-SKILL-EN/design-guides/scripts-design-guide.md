# Script File Design Guide

> Purpose: Define the security specifications, design methodology, and content requirements for executable scripts under the scripts/ directory
> Core principle: For code generated by an LLM and potentially auto-executed, security is the bottom line

---

## 1. Why a Script File Design Guide Is Needed

Scripts under the scripts/ directory are generated by an LLM and may be executed by automated systems. Unlike manually written code:
- LLMs may generate unsafe code (file deletion, network requests, system commands)
- Automated systems may execute without confirmation
- The execution environment may be production

If security specifications are not defined:
- Scripts may delete user data
- Scripts may leak sensitive information
- Scripts may execute malicious operations

Script files are **tool-type reference files** (L3), loaded on demand. When a SKILL needs to execute automated operations, the script file is loaded.

---

## 2. Script Security Specifications

### 2.1 Five Security Principles

| Principle | Description | Why It's Required |
|:---|:---|:---|
| **Read-Only First** | Scripts prioritize reading and do not modify the file system | Modification operations carry the highest risk; read-only operations are recoverable even on error |
| **Idempotency** | Multiple executions produce consistent results | Automated systems may repeat execution; non-idempotent scripts cause duplicate modifications |
| **Self-Contained** | No external environment dependencies | The execution environment is uncertain; external dependencies may cause failure or unintended operations |
| **Least Privilege** | Request only necessary permissions | The larger the privilege, the greater the risk; the least privilege principle limits the scope of potential harm |
| **Input Validation** | Validate all input, reject illegal input | LLM-generated input may contain malicious content; validation prevents injection attacks |

> Why five principles: Omitting any one makes script security unassured. Read-Only First is the minimum safety baseline; Idempotency ensures repeatable execution; Self-Contained ensures environment independence; Least Privilege limits harm scope; Input Validation prevents injection.

### 2.2 Prohibited Operations

- **Prohibited: File system modification**: Deleting, writing, renaming files/directories
- **Prohibited: Network requests**: HTTP/HTTPS requests, API calls, downloads
- **Prohibited: System command execution**: Shell commands, subprocesses, system calls
- **Prohibited: Environment variable modification**: Modifying PATH, modifying system configuration
- **Prohibited: Sensitive information access**: Reading passwords, keys, tokens, configuration files

> Why these operations are prohibited: These are "high-risk operations." LLM-generated scripts cannot guarantee the safety of these operations and must be prohibited.

### 2.3 Permitted Operations

- **File reading**: Read-only access, no modification
- **Regex matching**: Text scanning, pattern recognition
- **String processing**: Splitting, concatenation, formatting
- **Mathematical calculation**: Numeric operations, statistics
- **Data structure operations**: List, dictionary, set operations

> Why these operations are permitted: These operations are "read-only" or "pure computation" producing no side effects. Even if the script errors, it will not destroy data or leak information.

## 3. Generation Method (ur-skill Special Notes)

> This SKILL is a "SKILL for designing SKILLs" (ur-skill) and has no historical project accumulation.
> Therefore, scripts are not "collected from historical projects" but "dynamically designed based on the core scenarios of the generated SKILL."

### 3.1 Generation Based on Scenario Requirements

When using this SKILL to generate a specific SKILL, follow these steps to generate scripts:

1. **Determine script requirements**: Extract whether automated detection/validation/generation is needed from user requirements
   - Simple SKILL: Usually no script needed (no automation requirement)
   - Medium SKILL: May need 1-2 validation scripts
   - Complex SKILL: May need 3-5 scripts (scanning/validation/generation/conversion)

2. **Design script types**:
   - Scanning scripts: Detect anti-patterns, placeholders
   - Validation scripts: Verify format, constraints, compliance
   - Generation scripts: Batch generation, auto-fill (medium risk, human review required)
   - Conversion scripts: Format conversion, data migration (medium risk, human review required)

3. **Follow security specifications**:
   - [ ] Script safety level annotated (Safe / Medium Risk / High Risk)
   - [ ] Script read-only first, no high-risk operations
   - [ ] Script outputs to stdout, does not directly write files
   - [ ] Script has input validation and error handling

### 3.2 Generation Based on User Instructions

If the user explicitly requests script functionality:

1. **Analyze user requirements**: Clarify script type, input, output, execution environment
2. **Design script structure**: Design scripts following the template
3. **Security review**:
   - [ ] Script does not execute file deletion/writing/network requests/system commands
   - [ ] Script has SAFE_MODE toggle (default True)
   - [ ] Medium-risk and above scripts require human review

## 4. Script File Design Methodology

### 4.1 Script Types

| Type | Purpose | Safety Level | Example |
|:---|:---|:---:|:---|
| **Scanning Script** | Static check, issue identification | Safe | Anti-pattern scan, placeholder detection |
| **Validation Script** | Verify format, constraints, compliance | Safe | YAML frontmatter validation, table format check |
| **Generation Script** | Generate content, fill templates | Medium Risk | Batch SKILL file generation, auto-fill placeholders |
| **Conversion Script** | Format conversion, data migration | Medium Risk | Markdown to YAML, JSON to CSV |
| **Deployment Script** | Deploy to production | High Risk | Auto-deployment, CI/CD pipeline |

> Why safety levels: Scanning and validation scripts are read-only operations -- safe. Generation and conversion scripts may modify files -- medium risk. Deployment scripts directly operate on production environments -- high risk. High-risk scripts require human review and must not auto-execute.

### 4.3 Script Checklist

When writing scripts, check each item:

- [ ] Script has safety level annotation (Safe / Medium Risk / High Risk)
- [ ] Script has side effect description (None / Read / Write / Network)
- [ ] Script has input validation function
- [ ] Script does not execute high-risk operations (delete, write, network, system commands)
- [ ] Script outputs to stdout, does not directly write files
- [ ] Script has error handling (try/except)
- [ ] Script has logging (operation log for audit purposes)
- [ ] Script has passed human review (medium risk and above)
