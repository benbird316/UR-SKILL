# 故障诊断

> 用途：记录 UR-SKILL 执行过程中常见失败模式与修复动作
> 核心原则：每个故障配对"症状 → 根因 → 修复动作"（Troubleshooting 三元组），禁止纯声明式提示

---

## 1. 能力矩阵相关故障

### T01 能力矩阵写成工作流步骤

- **症状**：辐射领域的方向是"解析需求 → 检索资料 → 评估来源 → 生成报告 → 质量检查 → 交付输出"，看起来像流水线。
- **根因**：把工作流的执行顺序误当成独立的辐射领域，违反了排序测试和三问筛选。
- **修复动作**：
  1. 对候选领域执行排序测试与三问筛选（见 [templates/capability-architecture-template.md](../templates/capability-architecture-template.md)）。
  2. 不通过 → 退回调研步骤，重新推导真正的辐射领域。通过 → 保留。

### T02 辐射领域数量超过 8 个

- **症状**：能力矩阵行数 9+，信息密度下降，模型注意力稀释。
- **根因**：未做领域合并，或将工作流的子步骤错误地拆成了独立领域。
- **修复动作**：
  1. 检查哪些领域可以合并到更高层概念（如"结果验证"和"质量检查"→"质量保障"）。
  2. 对每个候选领域执行三问筛选（见 [templates/capability-architecture-template.md](../templates/capability-architecture-template.md)），淘汰不独立或重叠的领域。
  3. 将超出部分的细节下沉到 references/ 或合并为子能力。
  4. 目标：控制在 3-8 个辐射领域。

### T03 核心领域与辐射领域重叠

- **症状**：核心领域和某个辐射领域描述的是同一类能力，定义边界模糊。
- **根因**：核心任务边界不清，或辐射领域从核心领域拆分时未独立。
- **修复动作**：
  1. 重新定义核心领域——核心领域承载能力切面，辐射领域围绕核心提供独立补充能力。
  2. 若辐射领域与核心领域高度重叠 → 将辐射领域合并回核心，或重新定义核心以区分两者。
  3. 验证：核心领域 + 每个辐射领域应描述不同的能力域。

## 2. 工作流相关故障

### T04 关键节点检查项少于 6 维

- **症状**：调研（步骤 2）、架构（步骤 3）、校验（步骤 5）、验证（步骤 6）的检查清单只有 3-4 项，缺少某些审视维度。
- **根因**：误把关键节点当作非关键节点处理，未按节点类型分配审视维度。
- **修复动作**：
  1. 确认该步骤的节点类型：调研/架构/校验/验证 → 关键节点 → 6 维全开。
  2. 补齐缺失的检查项：目标对齐、事实锚定、方向校准、对抗验证、盲区识别、影响推演。
  3. 6 维必须全部出现，不可跳过。

### T05 盲区声明替代补齐动作

- **症状**：盲区识别只写"已标注局限""注意潜在盲区"，没有后续的调查或请求资源动作。
- **根因**：未按盲区三层机制执行——直接从第一层跳到第三层（纯声明式盲区）。
- **修复动作**：
  1. 第一层：执行调查分析，尝试自行优化填补盲区。
  2. 第二层（第一层后仍有不足）：向用户请求资源补充（如"请提供该领域的参考资料"）。
  3. 第三层（前两层都无资源补充）：输出盲区处理报告——已尝试动作 + 剩余盲区 + 可行性建议。
  4. 禁止纯声明式盲区（如"可能不知道"）——必须有已尝试动作。
  > 盲区三层机制详见 [templates/workflow-template.md](../templates/workflow-template.md)。

### T06 跳过未确认检查项

- **症状**：工作流步骤的某个检查项未确认（如盲区识别的盲区处理未填），但直接进入了下一步。
- **根因**：未遵循 Loop 循环原则——任一检查项未确认时，应先补齐再进入下一步。
- **修复动作**：
  1. 退回上一步，定位未确认的检查项。
  2. 执行对应的补齐动作（调查优化 / 请求资源 / 输出盲区报告）。
  3. 重新检查该项，确认后才进入下一步。
  4. 在交付前回溯全部检查清单，确保无遗漏。

## 3. 内容与格式相关故障

### T07 frontmatter 触发率不足

- **症状**：模型经常不触发该 SKILL，用户需要用精确措辞才能激活。
- **根因**：YAML frontmatter 的 `description` 字段过于保守或模糊，未覆盖同义触发词和多种触发场景。
- **修复动作**：
  1. 重写 `description`：前句必须回答"什么时候触发"，覆盖多种同义触发词。
  2. 包含领域名称和任务类型（如"Use when creating, designing, or packaging a SKILL.md file"而非"Use when designing SKILLs"）。
  3. `description` 字段限制 ≤1024 字符，但信息密度应高——去掉"when appropriate""as needed"等弱触发词。
  4. 验证：用不同的同义触发句式测试，确认都能触发。

### T08 引用的 references/ 文件不存在

- **症状**：SKILL.md body 中引用了 `references/examples.md`，但该文件缺失。
- **根因**：生成 SKILL 时遗漏了配套文件，或从其他 SKILL 复制了引用路径但未复制文件。
- **修复动作**：
  1. 检查 body 中所有 `references/` 开头的引用路径。
  2. 补齐缺失的参考文件（按对应的设计指南创建内容）。
  3. 若该文件不是必须的 → 从 body 中移除对应的引用声明。
  4. 运行 `scripts/validate_skill.py` 自动检测引用一致性。

### T09 body 行数超过 500 行

- **症状**：SKILL.md body 超过 500 行，中间区域信息被稀释（Lost in the Middle 效应）。
- **根因**：解释性内容、规范表格、示例未按渐进式加载下沉到 references/。
- **修复动作**：
  1. 将"为什么..."类解释内容迁移到 `design-rationale/design-rationale.md`。
  2. 将规范表格（字段约束、格式说明）下沉到 references/，body 只保留引用路径。
  3. 将示例迁移到 `references/examples.md`。
  4. 最终 body < 500 行（按 [design-guides/structure-guideline.md §4.3](../design-guides/structure-guideline.md) 的分段策略）。

### T10 交叉引用指向 UR-SKILL 内部文件

- **症状**：生成的 SKILL body 中出现 `templates/`、`design-guides/`、`References/`、`design-rationale/` 开头的路径引用。
- **根因**：生成 SKILL 时复制了 UR-SKILL 的引用路径，未替换为该 SKILL 自身的引用路径。
- **修复动作**：
  1. Grep 扫描生成 SKILL 的 body，匹配 `templates/`、`design-guides/`、`design-rationale/`、`References/` 模式。
  2. 发现的路径 → 替换为内联内容，或替换为该 SKILL 自身 references/ 中的文件路径。
  3. 交付前（步骤 7.3）再次 Grep 验证，确保零残留。

## 4. 质量与边界相关故障

### T11 风险边界被写成能力降级

- **症状**：风险边界声明中出现"不负责安全审查""只做意图识别"等措辞，像免责声明而非安全红线。
- **根因**：混淆了风险边界、专业边界和能力降级。风险边界应声明安全红线，而非推卸 SKILL 本应承担的能力。
- **修复动作**：
  1. 区分三类概念：风险边界 = 安全红线（触及即终止），专业边界 = 越界防护（不做超出专业范围的事），能力降级 = 反模式（做X但选择不报Y）。详见 [design-guides/boundary-design-guide.md](../design-guides/boundary-design-guide.md)。
  2. 检查风险边界声明中是否含"不负责""只做X不做Y""不属于"等能力降级措辞 → 删除。
  3. 确保每条风险边界声明至少有 1 个安全关键词（违法/公序良俗/歧视/攻击/注入/越狱/恶意）。

### T12 盲区声明格式不完整

- **症状**：盲区声明缺少"已尝试动作"字段，格式为"可能存在盲区，结果仅供参考，使用前请自行验证"。
- **根因**：盲区声明变成了免责条款（"甩锅"），而非按盲区三层机制驱动的能力提升手段。
- **修复动作**：
  1. 修改盲区声明为三层格式：已尝试动作（做了什么来填补盲区）+ 剩余盲区（仍未知什么）+ 可行性建议（建议怎么解决）。
  2. 删除"仅供参考""请自行验证""可能存在盲区""不保证"等甩锅措辞。
  3. 若确实是第三层（无资源补充），也必须先写清楚"已尝试动作"，然后才是"剩余盲区 + 可行性建议"。
  > 盲区三层机制详见 [templates/workflow-template.md](../templates/workflow-template.md)。

### T13 能力切面写成通用套话

- **症状**：能力切面写"掌握相关领域知识""识别潜在风险"——遮住内容后无法判断是做什么任务的。
- **根因**：未执行遮名测试——切面填充的是通用套话，对任何 SKILL 都成立，等于没写。
- **修复动作**：
  1. 对每个切面执行遮名测试：遮住切面内容，是否能推断出该 SKILL 的任务？
  2. 不通过 → 重写切面，加入该核心领域的具体知识栈（如工具名、规范名、漏洞类型）。
  3. 示例："掌握 Python 3.x 语法、PEP 8、OWASP Python Top 10"（通过）vs "掌握相关领域知识"（不通过）。

### T14 全局规则节点列表与步骤标题矛盾

- **症状**：执行中模型对某个步骤应用了意料之外的审视维度。例如，非关键步骤（3 维）却收到 6 维深度审查，浪费 token 并拖慢交付；或关键步骤只收到 3 维审查，跳过了强制质量门。
- **根因**：全局执行规则中的关键/非关键节点列表与实际步骤标题标签不同步。UR-SKILL 自身历史上，pre-analysis-engineer 的全局规则将"解析"列为关键节点（6 维），但解析步骤标题自身声明为非关键节点（3 维）——模型可能遵循最后读到的规则，产生不一致行为。
- **修复动作**：
  1. **定位矛盾**：将每个步骤标题的`【xxx节点，n 维】`标签与全局规则的节点列表交叉对照。找出在全局规则中被列为关键但步骤标题中声明为非关键的节点（或相反）。
  2. **以步骤标题为准**：步骤标题是真相来源——更新全局规则使其匹配。若步骤标题说非关键（3 维），全局规则必须将其列入非关键。
  3. **验证节点名匹配**：全局规则中每个节点名 MUST 在步骤标题中逐字出现（如主 SKILL 应为"前置分析"而非"解析"）。
  4. **重新校验**：修复后运行 `validate_skill.py` 确认各节点类型的检查项数量正确。

### T15 专业边界被写成能力降级

- **症状**：生成的 SKILL 专业边界声明中出现"只审查增量代码，不翻旧账""只做识别不做分析""不处理复杂场景"等措辞——运行时模型引用自身边界拒绝执行本该完成的任务，交付质量被边界声明自我阉割。
- **根因**：专业边界声明混淆了"操作归属"和"能力缩减"。专业边界应声明"我不做修复"（归属），而非声明"我只审增量不管存量"（降级）。后者实际上把本该执行的审查范围从边界声明中砍掉了。
- **修复动作**：
  1. **识别降级措辞**：扫描专业边界声明，找出"只做 X 不做 Y""不处理""不负责""仅限于"等能力缩减关键词。
  2. **还原为操作归属**：将能力降级改写为纯操作归属。如"只审查增量代码"→"不做代码修复或补丁编写"。归属不限制审查范围。
  3. **降级内容转盲区**：若确实存在超出能力范围的任务场景（如复杂场景），不应在边界中直接放弃，而应输出到盲区报告（追踪分析 + 根因 + 建议），继续交付其余部分。
  4. **对照检查**：修复后对照 [design-guides/boundary-design-guide.md](../design-guides/boundary-design-guide.md) 逐条确认专业边界不包含能力降级措辞。

### T16 生成的 SKILL 中存在占位符残留

- **症状**：生成的 SKILL 中包含未填充的占位符标记，如 `[TODO]`、`[FIXME]`、`[WIP]`、`TBD`、`___`、`...` 或空模板槽。运行时模型将这些解释为未定义行为区域，要么幻觉填充内容，要么静默跳过受影响章节，产生不完整的输出。
- **根因**：生成工作流完成步骤 4（执行）时未验证模板完整性即进入校验阶段。反模式 2 是最高优先级的扫描项，应阻断步骤 4 完成。
- **修复动作**：
  1. **扫描残留模式**：搜索生成的 SKILL 中的占位符标记（`[TODO]`、`[FIXME]`、`[WIP]`、`TBD`、`___`、连续省略号、空模板槽）。
  2. **决策：填充或删除**：对每个找到的占位符——若内容可确定，基于前置分析报告填充确定内容；若确实无法判断，删除整个章节，将其记录为盲区条目。
  3. **验证模板完整性**：对修复后的 SKILL 重新运行 `validate_skill.py`，确认零占位符残留。
  4. **对照检查**：对照 [spec-design-guide.md](../design-guides/spec-design-guide.md) 和 [metadata-spec.md](../templates/metadata-spec.md) 验证全部必填字段已填充。

### T17 生成代码脚本未执行可执行性验证

- **症状**：生成的 SKILL 包含代码脚本（如 validate_skill.py），但交付时未执行任何运行时测试。用户拿到脚本后首次运行即报错（语法错误、依赖缺失、逻辑缺陷），导致生成的 SKILL 整体可信度下降。
- **根因**：生成流程在步骤 6 验证阶段跳过了脚本可执行性检查。模型认为"代码看起来正确"即等于"可运行"，未将脚本执行和输出检查纳入交付门控条件。
- **修复动作**：
  1. **确认脚本清单**：扫描生成 SKILL 的 Scripts/ 目录，列出全部需要验证的脚本文件。
  2. **逐脚本执行**：对每个脚本执行 `python <script> --help` 或等价最小运行命令，确认无语法错误和导入错误。
  3. **输出正确性检查**：若脚本有标准输出（如 validate_skill.py 的校验结果），对照合法输入检查输出是否与预期一致。
  4. **重新校验**：全部脚本通过执行和输出检查后，重新运行 validate_skill.py 确认 SKILL 结构完整性未受修复影响。
