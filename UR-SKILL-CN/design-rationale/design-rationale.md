# 设计原理解释

> 用途：收容 UR-SKILL body 中的解释性内容，降低 body 信息密度
> 核心原则：body 只保留“做什么”和“怎么做”，所有“为什么”下沉到这里

---

## 1. 能力矩阵为什么是“能力域”而非“工作流步骤”

详见 ../templates/capability-architecture-template.md §1.1 — 排序测试、三问筛选（独立性/不可替代性/互补性）的完整定义。

简言之：能力矩阵回答“有什么能力”，工作流回答“按什么顺序执行”。把工作流步骤塞进能力矩阵是 反模式4（架构混淆）。

---

## 2. 为什么辐射领域数量由任务分析确定

辐射领域数量不固定，由任务分析确定（推荐 4-8 个，最少 3 个，最多 8 个）。

**确定方法**：
1. 分析核心任务：这个 SKILL 的核心任务是什么？
2. 识别核心领域：完成这个任务最核心的专业维度是什么？
3. 识别辐射领域：以核心任务为调查方向，调研完成该任务需要哪些关联专业领域。
4. 检查独立性：每个领域是否独立且互补？（排序测试 + 三问筛选）
5. 验证覆盖度：所有必要能力是否都被覆盖？
6. 控制数量：是否超过 8 个？如果超过，合并相关领域。

**为什么不是固定数量**：不同任务需要不同的能力领域。搜索分析需要 7 个辐射领域（搜索调研、信息管理、分析研究、深层推理、问题构建、成果表达、过程监控），Python 代码检查需要 5 个（静态分析、安全审计、性能诊断、风格规范、重构建议）。领域数量由任务分析确定，不是人为规定。

---

## 3. 为什么能力矩阵深度固定 4 层

4 层是能力递进的完整链条：

| 层级 | 认知水平 |
|:---|:---|
| 基础层 | 记忆 / 理解 |
| 进阶层 | 应用 |
| 高阶层 | 分析 / 评价 |
| 拓展层 | 创造 |

即使是简单任务，也需要完整的能力递进。没有深度的 SKILL 不如不用。

---

## 4. 为什么关键节点 6 维全开，非关键节点 3 维

6 个审视维度覆盖决策质量的完整要素：

| 审视项 | 问题 |
|:---|:---|
| 目标对齐 | 做了该做的事吗？ |
| 事实锚定 | 有依据吗？ |
| 方向校准 | 方向对吗？ |
| 对抗验证 | 经得起质疑吗？ |
| 盲区识别 | 有遗漏吗？识别后行动了吗？ |
| 影响推演 | 对后续有什么影响？ |

关键节点（调研、架构、校验、验证）是决策节点，需要充分审视。非关键节点（解析、执行、交付）是操作节点，路径已确定，只需确认目标、事实、盲区。

---

## 5. 为什么 body 限制 500 行 / 5000 tokens

基于 LLM 注意力机制：信息位于上下文两端时捕获率最高，中间显著下降（Lost in the Middle 效应）。超过阈值后，中间规则、约束、示例容易被稀释。

因此采用渐进式加载：
- L1：name + description（~100 tokens）
- L2：body（< 500 行 / < 5000 tokens）
- L3：references/、scripts/、assets/（按需加载）

---

## 6. 为什么规则使用 RFC 2119 关键词

RFC 2119（及 RFC 8174）是互联网工程标准，定义了 MUST / MUST NOT / SHOULD / SHOULD NOT / MAY 的精确语义。全大写时具有特殊含义，可被工具机械解析。

---

## 7. 为什么盲区识别后必须有具体行动

纯声明式盲区（"此处存在不足"）不驱动继续优化。三层机制强制行动：

1. 第一层：调查分析 → 自行优化填补
2. 第二层：仍有不足 → 请求资源
3. 第三层：无资源补充 → 输出盲区处理报告（已尝试动作 + 剩余盲区 + 可行性建议）

禁止从第一层直接跳到第三层。

## 8. 为什么复杂度判定基于资源需求而非领域数

复杂度判定基于**组织与维护需求**，与能力矩阵领域数**独立**，与参考文件数量**不直接挂钩**。

| 判定维度 | 简单 | 中等 | 复杂 |
|:---|:---|:---|
| **可执行脚本** | 不需要 | 不需要 | 需要 |
| **静态资源 / 模板** | 不需要 | 不需要 | 需要 |
| **迭代优化循环** | 单次产出 | 单次产出 | 需多轮验证/优化 |
| **工作流复杂度** | 4 步（解析→执行→校验→交付） | 5-6 步 | 7 步（含调研、架构设计） |
| **参考文件组织** | 无（单文件） | references/（至少含 example、anti-pattern、troubleshooting） | references/ + scripts/ + assets/ |
| **必备参考文件** | 无 | example + anti-pattern + troubleshooting（3 个底线） | 全部 8 类 design-guide 覆盖的文件 + 迭代脚本（具体文件类型由任务特征判定，见 §9 文件依赖决策） |
| **输出格式** | 单一格式 | 结构化格式 | 多格式混合 + 版本化 |

**复杂度决策树**（按以下顺序判定）：

1. **需要可执行脚本？** → 是 → **复杂**
2. **需要静态资源 / 模板？** → 是 → **复杂**
3. **需要多轮迭代优化（如通过 A/B 测试验证 Prompt 版本）？** → 是 → **复杂**
4. **需要分阶段工作流（调研→架构→执行→校验→验证→交付）？** → 是 → **中等**
5. **其余情况** → **简单**

### 模式 B（优化已有 SKILL）的复杂度判定补充

| 差距发现 | 复杂度影响 |
|:---|:---|
| 已有SKILL无能力矩阵 | → 至少中等（需要重新架构） |
| 已有SKILL无反模式/盲区三层/references/ | → 至少中等 |
| 已有SKILL无工具绑定 | → 至少中等 |
| 已有SKILL存在反模式10（能力降级） | → 至少中等 |
| 已有SKILL需要 scripts/ + assets/ | → 复杂 |

### 模式 C（知识提取）的复杂度判定补充

| 知识源特征 | 复杂度影响 |
|:---|:---|
| 单源（1个文档/1个数据库表） | → 简单或中等 |
| 多源（多个异构文档/多表跨库） | → 中等或复杂 |
| 需要多轮递归提取跨文档知识 | → 复杂 |

## 9. 为什么需要文件依赖决策树

复杂度判定完成后，进入文件依赖决策——确定该 SKILL 需要创建哪些 `references/`、`scripts/`、`assets/` 文件。此决策**在工作流步骤 3 结束后、步骤 4 开始前执行**，避免"执行到一半才发现漏文件"。

### 决策树（按顺序判定，每步 Yes 累加所需文件）

```
第 0 步：复杂度是否 ≥ 中等？
  ├── 否（简单） → 0 个 references/ 文件，所有内容在 body 中内联。结束。
  └── 是（中等/复杂） → 基础包：example + anti-pattern + troubleshooting（3 底线文件）
         ↓
 第 1 步：核心领域是否依赖外部知识库（如 OWASP/RFC/行业标准/法规/API 文档/学术理论体系）？
         ├── 是 → 需要 knowledge-reference 文件（K1-K4 子类型按知识来源判定）
         └── 否 → 跳过
         ↓
 第 2 步：是否自创方法论术语 ≥ 5 个，或涉及多领域交叉（如金融+法律、安全+合规）？
         ├── 是 → 需要 glossary.md
         └── 否 → 跳过
         ↓
 第 3 步：复杂度是否为"复杂"？
         ├── 否（中等） → 文件清单确定，结束。
         └── 是（复杂） → 额外需要 scripts/ + assets/，且：
               ├── 审查/审计/测试类任务 → 需要 output-content 规格（强制 Mermaid + 问题分级表）
               ├── 需要外部工具调用（API/CLI） → 需要 knowledge-reference K2（API 参考文件）
               ├── 需要静态模板产出（如报告、配置） → 需要 assets/templates/
               └── 需要可执行校验脚本 → 需要 scripts/validate_*.py
```

### 决策速查表

| 条件 | 触发文件 | 类型编号 | 强制/可选 |
|:---|:---|:---:|:---:|
| 中等+复杂度（门槛条件） | example.md | C1 | **强制** |
| 中等+复杂度（门槛条件） | anti-patterns.md | A1 | **强制** |
| 中等+复杂度（门槛条件） | troubleshooting.md | A2 | **强制** |
| 依赖外部知识库 | knowledge-reference 文件 | B1-B4 | 触发即强制 |
| 自创术语 ≥5 或多领域交叉 | glossary.md | D3 | 触发即强制 |
| 复杂 + 审查/审计/测试类 | output-content 规格 | A3 衍生 | 强制 |
| 复杂 + 外部工具调用 | K2 API 参考文件 | B2 | 推荐 |
| 复杂（门槛条件） | scripts/ + assets/ | — | **强制** |
| 需要自定义规范（如非标准 metadata-spec） | 规范型文件 | D1 | 按需 |

### 决策示例

**示例 1：Python 安全审计 SKILL**
```
0. 复杂度 ≥ 中等？→ 是（复杂，7 步工作流） → 3 底线
1. 依赖外部知识库？→ 是（OWASP Top 10 + CWE Top 25） → + knowledge-reference K1
2. 自创术语 ≥5 或多领域交叉？→ 是（安全领域术语） → + glossary
3. 复杂 + 审计类？→ + output-content 规格（强制 Mermaid + 问题分级），+ scripts/validate_*.py
```
→ 清单：example + anti-patterns + troubleshooting + knowledge-reference(OWASP) + glossary + output-content 规格 + scripts/

**示例 2：React 组件生成 SKILL**
```
0. 复杂度 ≥ 中等？→ 是（中等，5 步工作流） → 3 底线
1. 依赖外部知识库？→ 是（React API 参考 + 组件模式） → + knowledge-reference K2(API) + K4(patterns)
2. 自创术语 ≥5 或多领域交叉？→ 否
3. 中等 → 结束
```
→ 清单：example + anti-patterns + troubleshooting + knowledge-reference(React API + patterns)

**示例 3：博客写作指导 SKILL**
```
0. 复杂度 ≥ 中等？→ 是（中等） → 3 底线
1. 依赖外部知识库？→ 否（写作知识可内联）
2. 自创术语 ≥5 或多领域交叉？→ 否
3. 中等 → 结束
```
→ 清单：example + anti-patterns + troubleshooting（仅 3 底线文件）

## 10. 知识域设计论证

### 10.1 问题：旧辐射领域的反模式化

前置分析工程师的旧辐射领域曾为：

```
A 语义解析 → B 任务领域调研 → C 能力域推导 → D 复杂度判定 → E 文件依赖决策 → F 输出结构化
```

问题是这 6 个领域与工作流步骤**严格一一对应**（A→Step1, B→Step2, C→Step3, D+E→Step4, F→Step7），违反了 UR-SKILL 核心规则：**能力域 ≠ 工作流步骤**。

三问筛选不通过：
- **排序测试**：A→B→C→D→E→F 只能线性串行，重排后逻辑崩坏
- **独立性**：C"能力域推导"和 D"复杂度判定"在工作流中是序贯依赖的，无法独立发挥作用
- 结论：这是**反模式 4（工作流别名）**的经典案例，已写入 [../References/anti-patterns.md](../References/anti-patterns.md)

### 10.2 设计原则

前置分析工程师需要的是**真正的知识体**，而非工作流步骤的别名。它必须能像 HR 分析岗位一样，理解任何目标职业的专业内涵、独特风险和道德规范。

设计原则：
1. **通用职业视角**：前 4 域应能分析任何职业
2. **体系内化**：第 5 域坦诚承认需内化 UR-SKILL 自身方法论
3. **独立可交叉**：5 域可交叉调用，无时序依赖

### 10.3 6 域行业对标

| 知识域 | 行业对标 | 对标来源 |
|:---|:---|:---|
| 需求工程 | Requirements Engineering | IEEE SWEBOK v4.0 KA 2（Elicitation/Analysis/Specification/Validation） |
| 岗位能力分析 | Job Analysis / Competency Modeling | O\*NET KSAO 模型、I-O Psychology（知识/技能/能力/其他特质） |
| 能力域信息鉴评 | Information Quality Assessment / Media Literacy | CRAAP Test、SIFT Method、Lateral Reading（信息源鉴评与交叉验证） |
| 职业风险识别 | Professional Risk Assessment | HSE 5-step、ISO 31000（通用红线 + 职业特有风险谱系） |
| 职业道德规范 | Professional Ethics | APA Ethics Code、IEEE Code of Ethics、CFA Code of Ethics |
| 体系认知 | Institutional Knowledge | UR-SKILL 内部方法论（复杂度决策树/反模式/RFC 2119/检查清单） |

### 10.4 域选逻辑：前 5 域 vs 第 6 域

**前 5 域 = 对目标职业的外部认知**：

- **需求工程**：用户想做什么（目标/领域/交付形式/隐含假设）
- **岗位能力分析**：这个职业需要什么能力（KSAO 分析、能力域映射）
- **能力域信息鉴评**：支撑能力判断的信息是否可靠（官方标准 vs AI生成，时效性验证，信源谱系建立）
- **职业风险识别**：这个职业有什么独特风险（不只是"违法/歧视"，心理咨询有心理干预风险、金融有利益冲突风险、医疗有误诊风险）
- **职业道德规范**：这个职业的伦理约束（正向义务 + 负向约束，如心理咨询 MUST 保密，MUST NOT 利用心理弱势进行胁迫）

**第 6 域 = 对 UR-SKILL 的内部认知**：

- **体系认知**：内化全部方法论。这是唯一无法回避的内部知识，但坦诚命名，不做"通用技能"伪装。

### 10.5 与旧 6 域对比

| 维度 | 旧 6 域（反模式） | 新 6 域（正确） |
|:---|:---|:---|
| 领域本质 | 工作流步骤别名，线性串行 | 独立知识体，可交叉调用 |
| 三问筛选 | 不通过 | 通过 |
| 职业通用性 | 无（全是内部说明书） | 前 5 域可分析任何职业 |
| 信息质量 | 无过滤（搜到什么用什么） | 有能力域信息鉴评，过滤低质量/AI生成/过时信息 |
| 风险分析 | 仅面向反模式 | 可分析职业独特风险谱系 |
| 伦理约束 | 仅"触及安全红线→终止" | 可理解职业道德规范体系 |

### 10.6 职业风险：独特风险谱系（示例）

不同职业的风险谱系完全不同。以下为示例，实际分析时须根据目标职业展开：

| 职业 | 通用安全红线 | + 该职业的独特风险 |
|:---|:---|:---|
| 心理咨询 | 不违法/不歧视 | 不得利用心理弱势进行反向心理干预；不得建立双重关系 |
| 医疗诊断 | 不违法/不歧视 | 不得提供未经确诊的诊断结论；药物推荐须基于临床证据 |
| 金融投资 | 不违法/不歧视 | 不得承诺保本收益；须披露关联关系 |
| 法律咨询 | 不违法/不歧视 | 不得提供正式法律意见（无执照时）；须告知诉讼时效 |
| 教育辅导 | 不违法/不歧视 | 不得替代专业特殊教育评估；不得推荐未认证"治疗"方案 |

### 10.7 职业道德：双向结构

职业道德规范不是简单的"不能做X"清单，而是**双向结构**：

- **正向义务**：积极履行的职业责任（MUST 做什么）
- **负向约束**：禁止践踏的底线（MUST NOT 做什么）
- **执业边界**：区分本职业能做的和必须转介的

以 APA Ethics Code（心理咨询）示意：
```
正向义务：
  - MUST 保护来访者隐私与知情同意权
  - MUST 保持专业胜任能力，定期接受继续教育
  - MUST 在危机时采取合理保护措施

负向约束：
  - MUST NOT 利用心理弱势进行胁迫或控制
  - MUST NOT 在治疗关系外建立双重关系
  - MUST NOT 在社交媒体透露来访者信息

执业边界：
  - 能做：心理评估、咨询与治疗
  - 不能做：精神科药物处方（需转介精神科医生）
  - 不能做：司法精神鉴定（需专门资质）


## 11. 为什么 Step 1（解析）只做事实提取，Step 2（调研）才做深度分析

### 11.1 问题：旧设计的矛盾

旧版 pre-analysis-engineer Step 1 承担了过多认知负荷：

- Mode A Step 1.3：输出"目标 + 领域 + 交付形式"
- Mode B1 Step 1.2：初步差距分析（缺失项/缺陷项/可保留项）
- Mode C Step 1.3：映射为需求解析卡片（目标/领域/交付形式/隐含假设）

这些动作名义上是"解析"，实质上需要**深度调研和体系认知**才能完成。在未联网调查、未理解目标职业的专业内涵之前就输出"领域"和"差距判断"，逻辑倒置。

### 11.2 矛盾拆解

| 矛盾 | 症状 | 根因 |
|:---|:---|:---|
| 输出"领域"但未调研 | Step 1 声称产出领域归属，但对该职业的专业构成一无所知 | 解析应只提"用户说想做什么"，领域归属需联网调查才能确定 |
| B1"差距分析"即调研 | 缺失项/缺陷项/可保留项的判定需要深度体系认知 | 差距分析=发现"缺什么"，这本身就是调研的本质工作 |
| 域声明混乱 | `[WebSearch] 声明 岗位能力分析·高阶层` — 动作和域不匹配 | 域应声明动作所处的知识域，而非动作类型 |

### 11.3 正确边界

```
Step 1（解析）= 事实提取
  输入：用户自然语言
  输出：模式类型 + 用户核心需求摘要 + 暂定职业映射 + 隐含假设清单
         （B1: 原始结构卡片 / B2: 优化方向+事实摘要 / C: 知识源事实卡片）
  禁止：领域推导、差距分析、6切面审计、联网调查

Step 2（调研）= 深度认知
  输入：Step 1 的事实卡片
  输出：候选能力域（已通过排序测试+三问筛选）+ 职业风险谱系 + 职业道德规范 + 优化/补全方案
  动作：岗位分析 → 差距评估 → 能力推导 → 联网调查 → 审计 → 激活优化
```

### 11.4 类比：HR 招聘流程

```
Step 1（解析）= 收简历 + 读 JD
  → 看到：候选人写了"精通 Python"（事实）
  → 不判断：Python 是真精通还是只写过脚本（留给面试）

Step 2（调研）= 面试 + 技术评估
  → 验证：Python 理解深度（排序测试=问他能否教别人）
  → 发现：缺 Django 经验（差距分析=与岗位标准对比）
  → 决策：录用还是不录用（三问筛选=独立性/不可替代性/互补性）
```

### 11.5 设计原则总结

- **事实先于判断**：先知道"有什么"，再问"意味着什么"
- **解析不含调研**：解析=读，调研=想+查
- **域声明匹配动作**：`[WebSearch]` 声明的是该搜索服务于什么知识域，不是动作类型


## 12. UR-SKILL 主 SKILL 能力矩阵重设计

### 12.1 问题：旧 A-F 辐射领域的工作流别名化

UR-SKILL 主 SKILL 曾使用的 6 个辐射领域：

```
A 语义解析 → B 知识检索 → C 架构设计 → D 内容工程 → E 质量保障 → F 交付管理
```

与工作流步骤严格对应：

| 旧领域 | 工作流 |
|:---|:---|
| A 语义解析 | Step 1 解析 |
| B 知识检索 | Step 2 调研 |
| C 架构设计 | Step 3 架构 |
| D 内容工程 | Step 4 执行 |
| E 质量保障 | Step 5 校验 |
| F 交付管理 | Step 7 交付 |

不仅是一对一，命名本身就是动词（语义解析/知识检索/内容工程）——直接就是工作流步骤名。4 层递进（基础→进阶→高阶→拓展）强化了"步骤嵌套"感。

### 12.2 设计原则

同 pre-analysis-engineer（§10），UR-SKILL 的辐射领域必须是独立的知识体：

1. **基于 Prompt 工程师行业标准**：学术界（Vu & Oppenlaender 2025）+ 日本 PE-BoK（5 域）+ 行业技能矩阵
2. **名词非动词**：每个域是"懂什么"而非"做什么"
3. **无时序依赖**：任何工作流步骤都可调用任何域

### 12.3 新 6 域设计

| 知识域 | 行业对标 | 内涵 |
|:---|:---|:---|
| 需求工程与业务翻译 | SWEBOK Requirements Engineering / PE 沟通技能 | 模糊意图→结构化需求，映射业务目标到 SKILL 能力 |
| SKILL 架构设计 | PE 系统设计 / UR-SKILL 特有 | 能力矩阵/渐进式加载/复杂度决策树/文件依赖/工作流门控 |
| Prompt 体系工程 | 日本 PE-BoK 设计模式/结构化输出 | 信息架构（首因/近因）/RFC 2119/注意力管理/工具绑定语法 |
| 质量工程 | PE-BoK 迭代改进/评测 | 反模式检测/边界验证/完备性检查/交叉引用一致性 |
| 伦理安全 | PE-BoK 伦理与安全 / APA 伦理守则 | 风险边界（违法/歧视/注入）/职业特定风险谱系/专业边界 |
| 迭代改进 | PE 版本管理/平台适配 / HSE 5-step | 平台适配/盲区发现后自修复/元数据优化/版本演进 |
```

↓                                  Prompt 体系工程        质量工程
C 架构设计 → D 内容工程           伦理安全               迭代改进
↓
E 质量保障 → F 交付管理           ← 任何一步都可交叉调用任意域
