# 规范型文件设计指南

> 用途：定义规范型参考文件（metadata-spec.md, rules-template.md 等）的标准格式和设计方法
> 核心原则：规范型文件是"标准定义"，精确、无歧义、可验证

---

## 1. 为什么需要规范型文件设计指南

规范型文件定义标准格式（frontmatter 字段、规则关键词、表格结构）。与知识型文件不同：
- 规范型文件是**约束定义**，不是知识传递
- 规范型文件需要**精确无歧义**，不能模糊
- 规范型文件需要**可验证**，能自动化检查

如果不定义规范型文件：
- 生成的 SKILL 格式不一致
- frontmatter 解析失败
- 规则表述混乱，无法执行

规范型文件是**规范型参考文件**（L3），按需加载。当生成 SKILL 时，加载规范型文件确保格式合规。

---

## 2. 规范型文件的标准格式

### 2.1 三个必要要素

| 要素 | 说明 | 为什么必须 |
|:---|:---|:---|
| **字段约束表** | 每个字段的名称、类型、必填、范围、示例 | 精确约束，无歧义；自动化验证的基础 |
| **格式验证规则** | 如何验证格式合规 | 可自动化检查，降低人工审核成本 |
| **版本兼容性说明** | 版本变更记录，兼容性要求 | 规范演进时，使用者知道哪些变更需要适配 |

> 为什么三个要素：字段约束表定义"是什么"，格式验证规则定义"怎么检查"，版本兼容性说明定义"怎么演进"。少于三个，无法完整定义规范；多于三个，信息密度下降。

### 2.2 格式示例

```markdown
## 字段规范

| 字段 | 必填 | 类型 | 约束 | 示例 |
|:---|:---:|:---|:---|:---|
| name | MUST | string | <= 64 字符，kebab-case，a-z/0-9/- | python-code-inspector |
| description | MUST | string | <= 1024 字符，"Use when..." 开头 | Use when reviewing Python code... |
| type | MUST | string | 固定 prompt | prompt |

## 格式验证规则

- **MUST** 验证 name 符合 kebab-case：正则 `^[a-z0-9]+(-[a-z0-9]+)*$`
- **MUST** 验证 description <= 1024 字符：`len(description) <= 1024`
- **MUST** 验证 frontmatter 位于文件最开头，--- 前无空行

## 版本兼容性

| 版本 | 变更内容 | 兼容性 |
|:---|:---|:---:|
| v1.0 | 初始版本 | — |
| v1.1 | 增加 compatibility 字段 | 向后兼容 |
| v2.0 | 删除 type 字段，改为自动推断 | 不兼容，需升级 |
```

## 3. 生成方法（ur-skill特殊说明）

> 本 SKILL 是"设计 SKILL 的 SKILL"（ur-skill），自身没有历史项目积累。
> 因此，规范型文件不是"从历史规范收集"，而是"基于生成的 SKILL 的核心场景，动态调研生成"。

### 3.1 基于联网调查生成

当使用本 SKILL 生成具体 SKILL 时，按以下步骤生成规范：

1. **确定规范领域**：从用户需求中提取需要规范化的领域（如 frontmatter、API 接口、数据格式）
2. **联网调查行业标准**：搜索该领域的标准规范、最佳实践、官方文档
   - 例如：生成 REST API SKILL → 调研 OpenAPI 规范、RESTful API 设计最佳实践
   - 例如：生成 Python 代码 SKILL → 调研 PEP 8、Google Python Style Guide

3. **设计字段约束表**：
   - 字段名：基于行业标准命名
   - 必填/可选：基于使用频率确定
   - 类型：基于数据类型确定
   - 约束：基于行业标准确定（如 kebab-case、<= 64 字符）
   - 示例：基于常见用例确定

4. **验证规范**：
   - [ ] 每个字段有约束条件
   - [ ] 每个约束可验证（正则、计数、枚举、范围）
   - [ ] 每个约束有验证方法
   - [ ] 版本变更有兼容性说明

### 3.2 基于用户指示生成

如果用户提供了具体规范或已有标准：

1. **分析用户规范**：提取用户描述中的字段、约束、格式要求
2. **设计字段约束表**：将用户描述转化为"字段名 + 必填 + 类型 + 约束 + 示例"结构
3. **补充调研**：对用户未覆盖的字段，联网调查行业标准补充
4. **验证完整性**：
   - [ ] 覆盖用户提到的所有字段
   - [ ] 补充用户未提及但行业标准要求的字段
   - [ ] 约束可验证，无模糊表述

### 3.3 ur-skill 规范库

ur-skill 自身使用的规范型文件如下（这些文件是 ur-skill 的参考文件，不是生成的 SKILL 的参考文件）：

| 规范文件 | 规范内容 |
|:---|:---|
| ../templates/metadata-spec.md | YAML frontmatter 字段规范 |
| ../templates/rules-template.md | 规则格式规范（MUST/SHOULD/MAY） |
| ../design-guides/structure-guideline.md | 信息密度、注意力管理、正向表述 |
| ../templates/workflow-template.md | 工作流步骤、门控、驱动指令 |
| ../templates/capability-architecture-template.md | 能力矩阵、能力切面、三层关系 |
| ../design-rationale/design-rationale.md | 复杂度判定、渐进式加载、示例驱动 |
| ../templates/output-template.md | 输出结构、复杂度对应、格式约束 |

> 为什么ur-skill需要规范型文件：ur-skill设计 SKILL 时，需要统一的标准格式。这些规范型文件是ur-skill的"标准库"，确保所有生成的 SKILL 格式一致。

## 4. 规范型文件的设计方法

### 4.1 字段约束表设计

每个字段必须包含：

| 属性 | 说明 | 示例 |
|:---|:---|:---|
| **字段名** | 字段的标识 | name, description, type |
| **必填** | MUST / MAY（RFC 2119） | MUST = 必填，MAY = 可选 |
| **类型** | 数据类型 | string, number, boolean, array, object |
| **约束** | 精确约束条件 | <= 64 字符，kebab-case，固定值 |
| **示例** | 有效示例值 | python-code-inspector |

> 为什么五个属性：字段名标识字段，必填定义强制性，类型定义数据结构，约束定义合法范围，示例展示有效值。少于五个，约束不完整；多于五个，信息密度下降。

### 4.2 约束条件设计

约束条件必须是**可验证**的：

| 约束类型 | 验证方法 | 示例 |
|:---|:---|:---|
| **长度约束** | 字符计数 | `len(name) <= 64` |
| **格式约束** | 正则表达式 | `^[a-z0-9]+(-[a-z0-9]+)*$` |
| **值约束** | 枚举检查 | `type in ['prompt', 'skill', 'agent']` |
| **范围约束** | 数值比较 | `0 <= priority <= 100` |
| **结构约束** | 嵌套验证 | `metadata.updated` 必须符合 YYYY-MM-DD |

> 为什么可验证：不可验证的约束（如"格式良好"）是模糊约束，无法自动化检查。可验证的约束（如"正则匹配"）可被脚本自动化验证。

### 4.3 版本兼容性设计

版本变更必须标注兼容性：

| 兼容性 | 说明 | 处理方式 |
|:---|:---|:---|
| **向后兼容** | 新增字段，旧字段不变 | 旧 SKILL 无需修改 |
| **向前兼容** | 删除字段，或修改约束 | 旧 SKILL 必须升级 |
| **不兼容** | 字段重命名，或类型变更 | 旧 SKILL 必须重构 |

> 为什么标注兼容性：规范型文件会演进。不标注兼容性，使用者不知道是否需要升级已有 SKILL。

---

## 5. 规范型文件的内容要求

### 5.1 必须说明的内容

- **每个字段必须有约束条件**：不定义约束，字段值无法验证。
- **每个约束必须可验证**：不可验证的约束是模糊约束，无法自动化检查。
- **每个版本变更必须有兼容性说明**：不标注兼容性，使用者无法判断是否需要升级。

### 5.2 禁止的内容

- **禁止模糊约束**：如"格式良好""长度适中"，必须具体为"正则匹配""<= 64 字符"。
- **禁止无示例的约束**：没有示例，使用者不知道合法值长什么样。
- **禁止无版本控制的规范**：规范会演进，必须有版本号和变更记录。

---

## 6. 与模板文件的关系

规范型文件与模板文件的关系：

| 文件类型 | 职责 | 关系 |
|:---|:---|:---|
| **规范型文件**（metadata-spec.md） | 定义标准格式 | 源头：定义"应该怎么做" |
| **模板文件**（rules-template.md） | 引用规范型文件 | 消费：引用规范型文件中的约束 |
| **执行文件**（ur-skill.md） | 执行规范 | 执行：按规范型文件生成 SKILL |

> 为什么分层：规范型文件定义标准，模板文件引用标准，执行文件执行标准。规范型文件变更时，模板文件和执行文件同步更新。

## 7. 完整性检查清单

设计规范型文件时，逐项检查：

- [ ] 每个字段有五个属性（字段名、必填、类型、约束、示例）
- [ ] 每个约束可验证（正则、计数、枚举、范围、结构）
- [ ] 每个约束有验证方法（脚本可自动化检查）
- [ ] 每个版本变更有兼容性说明（向后兼容/向前兼容/不兼容）
- [ ] 无模糊约束（如"格式良好""长度适中"）
- [ ] 无无示例的约束
- [ ] 文件 < 200 行，超过则拆分
- [ ] 使用标准 Markdown 表格，无 ASCII 艺术线
