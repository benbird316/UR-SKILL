# 规范型文件设计指南

> 只教怎么写规范型参考文件。规范型文件是"标准定义"，精确、无歧义、可验证。
> 判定是否需要生成规范型文件，见 skill-package-design-guide.md §2。

---

## §1 标准格式

### 1.1 三个必要要素

| 要素 | 说明 | 为什么必须 |
|:---|:---|:---|
| **字段约束表** | 每个字段的名称、类型、必填、范围、示例 | 精确约束，自动化验证的基础 |
| **格式验证规则** | 如何验证格式合规 | 可自动化检查 |
| **版本兼容性说明** | 版本变更记录，兼容性要求 | 规范演进时，使用者知道哪些变更需要适配 |

### 1.2 格式示例

```markdown
## 字段规范

| 字段 | 必填 | 类型 | 约束 | 示例 |
|:---|:---:|:---|:---|:---|
| name | MUST | string | <= 64 字符, kebab-case | python-code-inspector |
| description | MUST | string | <= 1024 字符, "Use when..." 开头 | Use when reviewing... |
| type | MUST | string | 固定 prompt | prompt |

## 格式验证规则

- **MUST** 验证 name 符合 kebab-case: 正则 `^[a-z0-9]+(-[a-z0-9]+)*$`
- **MUST** 验证 description <= 1024 字符
- **MUST** 验证 frontmatter 位于文件最开头，--- 前无空行

## 版本兼容性

| 版本 | 变更内容 | 兼容性 |
|:---|:---|:---:|
| v1.0 | 初始版本 | — |
| v1.1 | 增加 compatibility 字段 | 向后兼容 |
| v2.0 | 删除 type 字段 | 不兼容，需升级 |
```

## §2 字段约束表设计

每个字段必须包含五个属性：

| 属性 | 说明 | 示例 |
|:---|:---|:---|
| **字段名** | 字段的标识 | name, description |
| **必填** | MUST / MAY（RFC 2119） | MUST = 必填，MAY = 可选 |
| **类型** | 数据类型 | string, number, boolean, array |
| **约束** | 精确约束条件 | <= 64 字符, kebab-case, 固定值 |
| **示例** | 有效示例值 | python-code-inspector |

---

## §3 约束条件设计

约束条件必须是**可验证**的：

| 约束类型 | 验证方法 | 示例 |
|:---|:---|:---|
| **长度约束** | 字符计数 | `len(name) <= 64` |
| **格式约束** | 正则表达式 | `^[a-z0-9]+(-[a-z0-9]+)*$` |
| **值约束** | 枚举检查 | `type in ['prompt', 'skill']` |
| **范围约束** | 数值比较 | `0 <= priority <= 100` |
| **结构约束** | 嵌套验证 | `metadata.updated` 符合 YYYY-MM-DD |

---

## §4 检查清单

- [ ] 每个字段有五个属性（字段名、必填、类型、约束、示例）
- [ ] 每个约束可验证（正则、计数、枚举、范围、结构）
- [ ] 每个约束有验证方法（脚本可自动化检查）
- [ ] 每个版本变更有兼容性说明（向后兼容/向前兼容/不兼容）
- [ ] 无模糊约束（如"格式良好""长度适中"）
- [ ] 无无示例的约束
- [ ] 文件 < 200 行，超过则拆分
