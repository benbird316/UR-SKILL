# 示例文件设计指南

> 用途：定义示例文件（examples.md）的标准格式、内容要求和设计方法
> 核心原则：示例是少样本学习（Few-Shot Learning）的核心，必须解释"为什么是这个输出"

---

## 1. 为什么需要示例文件

示例文件是 Prompt Engineering 的标准实践。LLM 通过示例学习任务模式，比纯指令更有效。

如果不提供示例：
- LLM 对输出格式的理解偏差大
- 边界场景处理不一致
- 错误场景无法识别

示例文件是**教学性参考文件**（L3），按需加载。当 LLM 生成 SKILL 内容时，加载示例文件理解期望输出格式。

---

## 2. 示例文件的标准格式

### 2.1 五个必要要素

| 要素 | 说明 | 为什么必须 |
|:---|:---|:---|
| **场景** | 什么情况下使用这个示例 | 让 LLM 理解示例的适用范围 |
| **Input** | 用户输入 | 定义输入格式和内容 |
| **Output** | 期望输出 | 定义输出格式和内容 |
| **Rationale** | 为什么是这个输出 | 教学核心；没有 Rationale，示例变成简单映射，LLM 无法推广到新场景 |
| **边界说明** | 为什么不是其他输出 | 边界/错误示例需要；让 LLM 理解什么情况下不能产生某个输出 |

> 为什么五个要素：LangChain FewShotPromptTemplate 要求 example_prompt 定义 input_variables 和 template。Rationale 对应"为什么是这个输出"的教学需求。少于五个，示例变成简单映射；多于五个，信息密度下降。

### 2.2 格式示例

```markdown
## 示例1：{场景描述}

**Input**：
```
{用户输入内容}
```

**Output**：
```
{期望输出内容}
```

**Rationale**：{为什么是这个输出，核心判断逻辑}

**边界说明**：{什么情况下不能产生这个输出，常见错误}
```

## 3. 生成方法（ur-skill特殊说明）

> 本 SKILL 是"设计 SKILL 的 SKILL"（ur-skill），自身没有历史项目积累。
> 因此，示例不是"从历史项目收集"，而是"基于生成的 SKILL 的核心场景，动态调研生成"。

### 3.1 基于联网调查生成

当使用本 SKILL 生成具体 SKILL 时，按以下步骤生成示例：

1. **确定核心场景**：从用户需求中提取 SKILL 的核心使用场景
2. **联网调查典型用例**：搜索该场景下的典型输入/输出对、教程、最佳实践案例
3. **设计示例**：
   - 典型场景：搜索最常见的 1-2 个用例
   - 边界场景：搜索边界条件、异常输入、极限值处理
   - 错误场景：搜索常见错误输入和错误输出模式
4. **验证示例**：
   - [ ] Input 完整无歧义
   - [ ] Output 完整无占位符
   - [ ] Rationale 解释清晰（为什么是这个输出）
   - [ ] 边界说明覆盖常见错误（为什么不是其他输出）

### 3.2 基于用户指示生成

如果用户提供了具体场景或已有材料：

1. **分析用户材料**：提取用户描述中的典型用例、边界情况、错误案例
2. **设计示例**：将用户描述转化为"Input → Output → Rationale → 边界说明"结构
3. **补充调研**：对用户未覆盖的场景，联网调查补充典型/边界/错误示例
4. **验证完整性**：
   - [ ] 覆盖用户提到的所有用例
   - [ ] 补充用户未提及但行业常见的边界场景
   - [ ] 示例数量与复杂度匹配（简单 1-2 个，中等 2-3 个，复杂 3 个以上）

### 3.3 ur-skill 生成示例

ur-skill 设计 SKILL 时，参考以下示例模板（详细生成流程见 ur-skill.md 工作流执行步骤）：

**示例1：生成简单 SKILL（Python 代码检查）**

**Input**：
```
用户需求：需要一个 SKILL 来检查 Python 代码质量
```

**Output**：
```yaml
---
name: python-code-inspector
description: "Use when reviewing Python code for quality issues..."
metadata:
  updated: 2026-07-09
  type: prompt
  whenToUse: 当需要审查Python代码质量时
---

# Python 代码检查器

## 能力矩阵
...
```

**Rationale**：简单 SKILL 只需要单文件，不需要 references/。能力矩阵核心领域 1 个 + 辐射领域由任务分析确定，检查维度按节点类型分配。

**边界说明**：如果用户要求检查多种语言，复杂度应提升为中等，增加 references/ 目录。

## 4. 示例文件的设计方法

### 4.1 按复杂度设计示例数量

| 复杂度 | 示例数量 | 覆盖要求 | 为什么 |
|:---|:---:|:---|:---|
| 简单 | 1-2 个 | 典型场景 | 简单任务模式单一，过多示例增加 token 消耗，边际收益递减 |
| 中等 | 2-3 个 | 典型场景 + 边界场景 | 中等任务有边界情况，需要展示边界处理能力 |
| 复杂 | 3 个以上 | 典型场景 + 边界场景 + 错误场景 + 复杂场景 | 复杂任务模式多样，需要覆盖错误场景让 LLM 学习"什么不能做" |

> 为什么数量与复杂度正相关：简单任务模式单一，1 个高质量示例足够。复杂任务模式多样，需要多个示例覆盖不同模式。但质量比数量更重要：1 个高质量示例 > 3 个低质量示例。

### 4.2 示例类型

| 类型 | 目的 | 必须包含 |
|:---|:---|:---:|
| **典型场景** | 展示标准处理能力 | 必须 |
| **边界场景** | 展示边界处理能力 | 中等及以上必须 |
| **错误场景** | 展示"什么不能做" | 复杂必须 |
| **复杂场景** | 展示多步骤/多条件处理能力 | 复杂必须 |

> 为什么需要错误场景：LLM 从错误示例中学习"什么不能做"，比从正面示例中学习"怎么做"更有效（负样本学习在机器学习中已被证明有效）。

### 4.3 设计示例的步骤

1. **识别典型场景**：最常见的 1-2 个使用场景
2. **识别边界场景**：输入接近边界值、条件模糊、多条件冲突
3. **识别错误场景**：常见错误输入、常见错误输出
4. **验证示例质量**：
   - [ ] Input 完整无歧义
   - [ ] Output 完整无占位符
   - [ ] Rationale 解释清晰
   - [ ] 边界说明覆盖常见错误

## 5. 示例文件的内容要求

### 5.1 必须说明的内容

- **每个示例必须有 Rationale**：没有 Rationale，示例变成简单 Input→Output 映射，LLM 无法推广到新场景。
- **边界/错误示例必须有"边界说明"**：不解释"为什么不是其他输出"，LLM 可能在类似场景下重复错误。
- **示例必须完整无占位符**：Output 中的 `[xxx]`、`{待填充}` 等占位符违反反模式2。

### 5.2 禁止的内容

- **禁止无 Rationale 的示例**：没有 Rationale 的示例是"黑盒映射"，LLM 无法学习推理过程。
- **禁止占位符残留**：示例 Output 必须完整，不能包含未填充内容。
- **禁止示例与 body 重复**：示例放入 ../examples/examples.md，body 只保留引用声明。

---

## 6. 与能力架构的协同

示例文件支撑能力架构中的能力层级：

| 能力层级 | 示例类型 | 作用 |
|:---|:---|:---|
| 基础层 | 典型场景 | 展示标准处理能力 |
| 进阶层 | 边界场景 | 展示边界处理能力 |
| 高阶层 | 错误场景 | 展示错误识别和修复能力 |
| 拓展层 | 复杂场景 | 展示多步骤/多条件处理能力 |

---

## 7. 完整性检查清单

设计示例文件时，逐项检查：

- [ ] 示例数量与复杂度匹配（简单 1-2，中等 2-3，复杂 3+）
- [ ] 每个示例包含五个要素（场景、Input、Output、Rationale、边界说明）
- [ ] 每个示例有 Rationale（为什么是这个输出）
- [ ] 边界/错误示例有边界说明（为什么不是其他输出）
- [ ] 示例 Output 完整无占位符
- [ ] 示例与 body 不重复（示例在 references/，body 只引用）
- [ ] 文件 < 200 行，超过则拆分
- [ ] 示例覆盖典型 + 边界 + 错误（中等及以上）
