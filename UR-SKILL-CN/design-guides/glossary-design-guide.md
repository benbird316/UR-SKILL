# 术语表文件设计指南

> 只教怎么写 references/glossary.md。术语表是体系内概念的一致性锚点，消除歧义、防止概念漂移。
> 判定是否需要生成术语表，见 skill-package-design-guide.md §2。

---

## §1 标准格式

### 1.1 四个必要要素

| 要素 | 说明 | 为什么必须 |
|:---|:---|:---|
| **术语** | 精确定义的名称 | 检索与引用的唯一标识 |
| **定义** | 单句精确定义，无歧义 | 核心；术语表的唯一价值 |
| **所属域** | 术语在哪个概念域中使用 | 同一术语可能在不同域中有不同含义 |
| **使用示例** | 在 SKILL 体系中的实际用法 | 桥接抽象与具体 |

### 1.2 格式模板

```markdown
## {术语}

- **定义**: {一句话精确定义，使用该域内的其他已定义术语}
- **所属域**: {该术语在哪个概念域中使用}
- **示例**: {在 SKILL body / references 中的实际用法，1-2 句}
```

### 1.3 组织方式

术语按**概念域分组**，域内按**逻辑依赖**排序（先定义基础术语，再定义依赖它的复合术语）。

常见概念域分组示例：

| 概念域类别 | 示例分组名 | 适用场景 |
|:---|:---|:---|
| 核心概念类 | 核心概念、系统实体 | 定义 SKILL 的基本术语 |
| 执行类 | 工作流步骤、执行阶段 | 定义流程相关术语 |
| 质量类 | 校验节点、质量标准 | 定义质量保证相关术语 |
| 输出类 | 输出格式、交付物 | 定义输出相关术语 |

### 1.4 编号系统

每个术语分配 **G{序号}** 编号（G = Glossary），便于跨文件引用。
如：
```markdown
## G01 能力矩阵
...
## G02 核心领域
...
```

## §2 内容要求

### 2.1 必须

- 每个术语必须有"定义"
- 定义使用该域内的已定义术语（优先引用编号术语）
- 每个术语必须有"示例"

### 2.2 禁止

- **MUST NOT** 循环定义（术语 A 依赖 B，B 依赖 A）
- **MUST NOT** 过度定义（如"检查清单是用于检查的清单"）
- **MUST NOT** 外部引用替代定义（如"详见某外部链接"）
- **MUST NOT** 模糊限定词（如"通常""一般""大多数情况下"）

---

## §3 检查清单

- [ ] 每个术语有四个要素（术语、定义、所属域、示例）
- [ ] 定义用一句话完成，无同义反复
- [ ] 定义之间无循环依赖
- [ ] 相近术语的定义能区分彼此
- [ ] 按概念域分组，域内按逻辑依赖排序
- [ ] 术语有 G{序号} 编号
- [ ] 术语数量由 research-analyst 调查分析结果推导
- [ ] 每个术语在体系中至少出现 2 次（非一次性用词）
- [ ] 无外部引用替代定义（自包含）
- [ ] 文件 < 200 行，超过则拆分
