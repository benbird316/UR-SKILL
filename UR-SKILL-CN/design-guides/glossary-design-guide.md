# 术语表文件设计指南

> 用途：定义术语表参考文件（glossary.md）的标准格式、内容要求与设计方法
> 核心原则：术语表是体系内概念的一致性锚点，消除歧义、防止概念漂移

---

## 1. 为什么需要术语表文件

术语表是 SKILL 体系的"语义协议"。当 SKILL 涉及领域专有概念、自定义方法论或抽象术语时，不同文件（SKILL.md body、references/ 各文件、templates/ 各模板）中的同一术语必须指向同一概念。

如果不维护术语表：
- 同一术语在不同文件中被不同理解（概念漂移）
- 新加入的 LLM 无法从上下文中推断术语精确定义
- 跨文件引用时产生"术语说了但没定义"的认知缺口
- 体系演进时术语语义变更无法追踪

**ur-skill 自身的特殊性**：ur-skill 是"设计 SKILL 的 SKILL"，它定义了一套方法论术语（能力矩阵、辐射领域、排序测试、三问筛选、盲区三层机制……）。这些术语分散在多个文件（SKILL.md body、templates/、design-guides/、References/），但没有统一的定义源。术语表是消除这种分散性的唯一手段。

术语表文件是**知识型参考文件**（L3），按需加载。当 LLM 在 SKILL.md body 或 references/ 中遇到不确定的术语时，加载术语表获取精确定义。

## 2. 术语表的标准格式

### 2.1 四个必要要素

| 要素 | 说明 | 为什么必须 |
|:---|:---|:---|
| **术语** | 精确定义的名称 | 检索与引用的唯一标识 |
| **定义** | 单句精确定义，无歧义 | 核心；术语表的唯一价值在于"这个词到底什么意思" |
| **所属域** | 术语在哪个概念域中使用 | 同一术语可能在不同域中有不同含义（如"门控"在工作流域和交付域） |
| **使用示例** | 在 SKILL 体系中的实际用法 | 定义抽象 → 示例具体，"看懂了但不知道怎么用"是术语表失效的常见原因 |

> 为什么四个要素：术语标识检索目标，定义消除歧义，所属域防止跨域混淆，使用示例桥接抽象与具体。少于四个，术语表变成"关键词列表"而非"语义锚点"；多于四个，信息密度下降。

### 2.2 格式模板

```markdown
## {术语}

- **定义**：{一句话精确定义，使用该域内的其他已定义术语}
- **所属域**：{该术语在哪个概念域中使用，如"工作流域""架构域""质量域"}
- **示例**：{在 SKILL body / references 中的实际用法，1-2 句}
```

### 2.3 组织方式

术语按**概念域分组**，域内按**逻辑依赖**排序（先定义基础术语，再定义依赖它的复合术语）。

| 排序策略 | 适用场景 | 为什么不适用 |
|:---|:---|:---|
| 字母序排列 | 大型通用词典（100+ 术语） | SKILL 术语表通常 20-50 个术语，字母序割裂概念关联 |
| **概念域分组 + 逻辑依赖** | SKILL 体系术语表 | 术语之间有依赖关系——必须先理解"能力矩阵"才能理解"辐射领域" |

> 为什么按概念域分组：术语表的使用场景是"遇到不懂的词来查"，但 SKILL 术语表的使用场景更接近"理解整个体系的概念地图"。按域分组帮助 LLM 建立概念间的关联，而非孤立记忆。

### 2.4 编号系统

每个术语分配 **G{序号}** 编号（G = Glossary），便于跨文件引用。

```markdown
## G01 能力矩阵
...
## G02 核心领域
...
```

> 为什么编号：与反模式编号（反模式1）、规则编号（规则01）一致，遵循 UR-SKILL 的编号系统规范（见 structure-guideline.md §1.2）。

## 3. 术语表的内容要求

### 3.1 必须说明的内容

- **每个术语必须有"定义"**：不定义，术语表是术语列表而非术语表。
- **定义必须使用该域内的已定义术语**：用未定义术语解释定义 = 循环依赖。优先引用术语表中已有的编号术语。
- **每个术语必须有"示例"**：不示例，"看懂了但不知道怎么用"。

### 3.2 禁止的内容

- **禁止循环定义**：术语 A 的定义依赖术语 B，术语 B 的定义依赖术语 A。
- **禁止过度定义**：如"检查清单是用于检查的清单"——什么都不解释。
- **禁止外部引用替代定义**：如"详见某外部链接"——术语表是自包含的语义锚点，不能指望 LLM 去外部查定义。
- **禁止模糊限定词**：如"通常""一般""大多数情况下"——术语定义必须是确定的。

### 3.3 定义质量标准

| 标准 | 说明 | 反例 → 正例 |
|:---|:---|:---|
| **单句定义** | 定义用一句话完成，不超过 50 字 | "能力矩阵是一个矩阵，包含核心领域和辐射领域……" → "能力矩阵：1 个核心领域 + 3-8 个辐射领域 × 4 层深度的能力结构化表达" |
| **无同义反复** | 定义不重复术语本身 | "辐射领域是核心领域的辐射领域" → "辐射领域：围绕核心领域的独立专业能力维度" |
| **区分相近概念** | 定义能区分易混淆术语 | "风险边界"和"专业边界"的定义必须能让人区分两者 |

## 4. 生成方法（ur-skill 特殊说明）

> 本 SKILL 是"设计 SKILL 的 SKILL"（ur-skill），自身需要术语表；同时也为生成的 SKILL 提供术语表设计能力。

### 4.1 为 ur-skill 自身创建术语表

按以下步骤：

1. **提取候选术语**：扫描 SKILL.md body、templates/、design-guides/、References/ 中出现的专有概念和方法论术语
2. **去重归类**：合并同义词，按概念域分组
3. **按逻辑依赖排序**：基础概念 → 复合概念 → 方法论概念
4. **验证完整性**：
   - [ ] 每个术语在至少 2 个文件中出现（确保不是一次性用词）
   - [ ] 每个术语有定义、所属域、示例
   - [ ] 定义之间无循环依赖
   - [ ] 相近术语的定义能区分彼此

### 4.2 为生成的 SKILL 创建术语表

当生成的 SKILL 包含领域专有术语时（如 Python 代码审计 SKILL 的 "PEP 8""CWE""OWASP Top 10"），在 `references/glossary.md` 中创建术语表：

1. **从核心领域提取**：核心领域是什么？辐射领域是什么？每个领域的关键术语有哪些？
2. **从示例中提取**：examples.md 的 Input/Output 中出现了哪些专有术语？
3. **从反模式中提取**：anti-patterns.md 中涉及的概念是否有需要定义的？
4. **控制数量**：术语表 10-30 个术语，按概念域分 2-4 组。超过 30 个，信息密度下降。

### 4.3 术语表的触发条件

术语表是否需要，不由复杂度单独决定，而由 design-rationale.md §9 文件依赖决策树的第 2 步判定：

> **"是否自创方法论术语 ≥ 5 个，或涉及多领域交叉（如金融+法律、安全+合规）？"**
> — 是 → 需要 glossary.md（触发即强制）

复杂度与术语表的参考关系（仅供参考，最终由决策树判定）：

| 复杂度 | 术语表 | 理由 |
|:---|:---|:---|
| 简单 | 不需要 | 简单 SKILL 的概念少，自创术语通常 < 5，术语在 body 中内联解释 |
| 中等 | 可能（取决于术语数） | 中等 SKILL 的方法论术语若 ≥ 5 个 → 需要；< 5 个 → 可选 |
| 复杂 | 通常需要 | 复杂 SKILL 几乎必然自创术语 ≥ 5 或因多领域交叉而需要 |

## 5. 术语表与体系其他文件的关系

| 文件 | 关系 | 说明 |
|:---|:---|:---|
| **SKILL.md body** | body 引用术语编号 | body 中首次出现的术语标注编号引用：`能力矩阵（G01）` |
| **references/anti-patterns.md** | 反模式描述引用术语编号 | 反模式的"表现"和"危害"中引用术语编号确保概念一致 |
| **references/troubleshooting.md** | 故障诊断引用术语编号 | 故障症状描述中使用术语编号 |
| **templates/** | 模板引用术语编号 | capability-architecture-template.md 中的概念引用术语表编号 |

> 为什么跨文件引用术语编号：确保体系内所有文件对同一概念的理解一致。编号引用比内联定义更高效——LLM 看到编号就知道去术语表查精确定义，避免在 body 中重复定义。

## 6. 完整性检查清单

设计术语表文件时，逐项检查：

- [ ] 每个术语有四个要素（术语、定义、所属域、示例）
- [ ] 定义用一句话完成，无同义反复
- [ ] 定义之间无循环依赖
- [ ] 相近术语的定义能区分彼此
- [ ] 按概念域分组，域内按逻辑依赖排序
- [ ] 术语有 G{序号} 编号
- [ ] 术语数量与复杂度匹配（中等 10-20，复杂 20-30）
- [ ] 每个术语在体系中至少出现 2 次（非一次性用词）
- [ ] 无外部引用替代定义（自包含）
- [ ] 文件 < 200 行，超过则拆分
