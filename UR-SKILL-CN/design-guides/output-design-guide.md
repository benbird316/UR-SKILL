# 输出内容设计指南

> 只教怎么设计 SKILL 的输出。回答"输出什么、什么形态、怎么给到用户"。
> 判定是否需要输出内容设计，见 skill-package-design-guide.md §2。

---

## §1 设计思维：4 问框架

在动手设计输出之前，逐项回答以下 4 个问题。

### Q1: 输出形式 — 产物是什么形态？

从以下 6 个高层类别中选择（具体 11 种格式武器见 §2）：

| 类别 | 适用场景 |
|:---|:---|
| 纯文本 / Markdown | 分析报告、叙事内容、创意讨论 |
| 结构化表格 | 问题清单、评分卡、对比分析 |
| Mermaid 图表 | 流程可视化、状态转换、组件关系 |
| 代码块 | 代码生成、配置输出、脚本 |
| JSON/YAML 结构化数据 | 需要下游系统解析的输出 |
| 混合 | 大多数 SKILL——文本+表格+代码块 |

不确定时，默认 `混合（Markdown 正文 + 结构化表格 + 代码块）`。

### Q2: 输出结构 — 按什么顺序组织？

通用模式：**结论先行 → 主体展开 → 建议收尾**。

```
1. 结论/摘要    — 让读者 5 秒内知道结果
2. 主体内容     — 详细分析、数据、证据
3. 正面反馈     — 不只有负面（适用时）
4. 建议/下一步  — 可执行的动作项
```

每个区块声明"内容包括什么"，而非"本节写那个主题"。

### Q3: 质量标准（Win Metric）— 什么算"好"？

来自 FLOW 框架的核心洞察：定义输出必须通过的**真实世界测试**——而非抽象形容词。

| SKILL 类型 | Win Metric 示例 |
|:---|:---|
| 代码审查 | "一个初级开发者能根据每一条评论独立修复，不需追问" |
| 安全审计 | "运维团队能根据报告直接判断是否需要紧急回滚" |
| 小说大纲 | "用户只需一次追问即可进入创作，不需要再次解释设定" |
| 聊天陪伴 | "连续 5 轮对话不出现风险内容，且用户未因语气不适中断对话" |
| 数据分析 | "CFO 能在 3 分钟内看完并做出是/否决策" |

每条 Win Metric 必须是具体的、可验证的。禁止"高质量"、"优秀"等空洞词汇。

**分级失败模式**（当输出达不到 Win Metric 时）：

| 失败类型 | 适用 SKILL | 策略 |
|:---|:---|:---|
| 功能型 | 代码审查、安全审计、数据分析 | 定义精确量化标准（如"漏洞检出率 > 85%"） |
| 创意型 | 小说创作、内容生成 | 给出风格一致性标准（如"无角色崩坏"、"语气连贯"） |
| 社交型 | 聊天陪伴、情感支持 | 给出安全/得体标准（如"无歧视性表达"、"不越界提供心理咨询"） |

### Q4: 交付模式 — 怎么给到用户？

4 种模式，按 SKILL 性质选择：

| 模式 | 适用场景 | 说明 |
|:---|:---|:---|
| **一次性输出** | 查询、生成、翻译 | 一次交互完成 |
| **确认-修复-验证循环** | 审查、诊断、迭代改进 | 最多 N 轮，每轮用户确认/修改后重新验证 |
| **分阶段交付** | 架构设计、多步骤规划 | Phase 1 → 确认 → Phase 2 |
| **交互式对话** | 咨询、陪伴、创意讨论 | 无固定轮数，基于对话流自然推进 |

循环模式必须注明循环上限。推荐注明退出口令（如"用户说'不用了'则结束"）。

## §2 格式类型分类学

先认识有什么武器，再决定用什么武器。

| 格式类型 | 适用场景 | 优点 | 缺点 |
|:---|:---|:---|:---|
| **Markdown 表格** | 数据对比、问题清单、分级结果 | 结构清晰，可排序 | 无法表达流程逻辑 |
| **Mermaid 流程图** | 业务流程、调用链、数据流 | 流程与关系一目了然 | 静态图，无法交互 |
| **Mermaid 序列图** | API 调用链、请求响应流 | 时序关系清晰 | 节点多时冗长 |
| **Mermaid 状态图** | 状态机、工作流转换 | 状态转换可视化 | 仅适用状态驱动场景 |
| **代码块** | 修复方案、示例代码 | 可直接复制 | 必须标注语言 |
| **检查清单** | 验证项、自检项、任务列表 | 逐项可验证 | 缺乏上下文解释 |
| **执行摘要** | 高层结论、决策建议 | 快速阅读 | 缺乏细节 |
| **严重度分级表** | 安全漏洞、质量问题 | 优先级明确 | 需准确定义等级 |
| **对比矩阵** | 方案选型、工具对比 | 多维度直观对比 | 维度增加时变复杂 |
| **叙事报告** | 分析结论、研究调查 | 上下文丰富 | 冗长，难以快速定位 |
| **判决结论** | 给出明确行动决策 | 直接可操作 | 二元判断可能过于简化 |

## §3 可视化规则

### 3.1 强制条件（满足任一即触发）

1. 输出涉及**流程/逻辑链**描述
2. 输出涉及**多组件交互**
3. 输出涉及**状态转换**

### 3.2 强制规则

| 规则 | 说明 |
|:---|:---|
| 最少 1 张图 | 触发强制条件时，必须包含至少 1 个 Mermaid 图表 |
| 优先流程图和序列图 | `flowchart` 或 `sequenceDiagram`，展示过程/调用/数据流 |
| 禁止纯分类框图 | 只做分组无流向关系的框图不允许 |
| 强制配色 | 填充色和文字色必须显式指定，确保明暗主题都可读 |
| 图表类型匹配 | 架构→`graph TD`，调用→`sequenceDiagram`，状态→`stateDiagram` |

### 3.3 豁免条件

- 输出内容为纯文本分析（无流程关系）
- 输出内容为单一结论（如"通过/不通过"）

## §4 输出结构通用模式

### 4.1 结论/摘要层

```markdown
## 结论

**核心判断**: {一句话结论}
**关键发现**:
1. [最重要的发现]
2. [次重要发现]
```

### 4.2 主体内容层

按逻辑分组展开，格式由 §2 的武器选择决定。每个分组声明"这里给出什么"而非"这里叫什么"。

### 4.3 正面反馈（推荐）

```markdown
## 正面观察

- ✅ [值得肯定的点]
- ✅ [做得好的地方]
```

### 4.4 建议/下一步

```markdown
## 建议

### 优先执行
- [ ] [行动项]

### 后续考虑
- [ ] [行动项]
```

## §5 用户交互模式

| 模式 | 适用场景 | 示例 |
|:---|:---|:---|
| **一次性报告** | 简单查询、研究调查 | 输出完整报告后结束 |
| **确认-修复-验证循环** | 审查、诊断类 | 输出问题 → 用户确认 → 修复验证 → 确认 |
| **分阶段交付** | 复杂分析 | 先出摘要 → 用户确认 → 出详细报告 |
| **异步定时** | 周期性巡检 | 定时执行 → 自动推送到指定位置/频道 |

## §7 规则

- 分级标准必须量化——禁止"严重"、"重要"等模糊词，必须写触发条件
- 交付清单每项必须可二元验证（是/否）——禁止"确保质量"等模糊措辞
- 输出结构按用户阅读顺序排列（结论 → 主体 → 建议）
- Win Metric 必须具体可验证，禁止空洞形容词
- 交付模式必须注明循环上限或退出口令
- 可视化触发时必须包含 Mermaid 图表，颜色显式指定
