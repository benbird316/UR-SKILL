# 输出内容设计指南

> 用途：指导 UR-SKILL 在生成 SKILL 时，定义输出内容的**格式类型、可视化要求、结构层次、交互策略、判决策略**。回答"SKILL 执行后到底输出什么"这个问题。

---

## 1. 为什么需要输出内容设计

**问题一：输出形式单一**

当前 ../templates/output-template.md 只定义了 Markdown 文件的骨架（能力矩阵 + 工作流 + 规则），但没有定义**运行时输出**——Agent 执行 SKILL 后呈现给用户的结果长什么样。

以 TRAE-code-review 为例，它在 SKILL.md 的 Step 4 中强制要求：

> **Always provide** at least one mermaid diagram to summarize the key changes.

但 quality 和 security SKILL 完全没有这种规格要求——它们只定义了 SKILL 自身的结构模板，没定义 SKILL 被 Agent 使用时的输出内容格式。

**问题二：输出类型无决策矩阵**

generate 生成 SKILL 时，它不知道什么时候该让 SKILL 输出 Mermaid 图表 vs 表格 vs 检查清单 vs 叙事报告。所有 SKILL 一律输出 Markdown 正文，缺乏根据任务类型匹配输出格式的能力。

**解决方案**：本指南定义输出内容的设计维度——格式类型、可视化强制规则、结构层次规范、判决策略、用户交互模式。

---

## 2. 输出格式类型全景

### 2.1 类型分类

| 格式类型 | 适用场景 | 优点 | 缺点 |
|:---|:---|:---|:---|
| **Markdown 表格** | 数据对比、问题清单、分级结果 | 结构清晰、可排序 | 无法表达过程逻辑 |
| **Mermaid 流程图** | 业务流程、调用链、数据流 | 一眼看懂过程和关系 | 静态图表，无法交互 |
| **Mermaid 序列图** | API 调用链、请求-响应流 | 时序关系清晰 | 节点多时过长 |
| **Mermaid 状态图** | 状态机、工作流转换 | 状态迁移可视化 | 仅适用于状态驱动场景 |
| **代码块** | 展示修复方案、示例代码 | 可直接复制 | 需标注语言 |
| **检查清单** | 验证项、自检项、任务列表 | 逐项可核对 | 无上下文解释 |
| **执行摘要** | 高层结论、决策推荐 | 快速阅读 | 缺少细节 |
| **严重度分级表** | 安全漏洞、质量问题 | 优先级明确 | 需准确定义等级 |
| **对比矩阵** | 方案选型、工具比较 | 多维度直观对比 | 维度增多时复杂 |
| **叙事报告** | 分析结论、研究调查 | 上下文丰富 | 冗长，难快速定位 |
| **判裁决断** | 合入/拒绝/修改 | 给出明确行动决定 | 二元判断可能过于简化 |

### 2.2 格式选择决策矩阵

| 任务类型 | 首选格式 | 强制可视化 | 注释 |
|:---|:---|:---:|:---|
| **代码审查** | 问题表格 + Mermaid 流程图 | ✅ 是 | 流程图展示变更的业务流和技术流 |
| **安全审计** | 严重度分级表 + 执行摘要 + 代码块 | ❌ 可选 | 表格按严重度降序，代码块展示漏洞位置 |
| **架构评审** | Mermaid 架构图 + 对比矩阵 | ✅ 是 | 架构图展示组件关系，矩阵对比方案 |
| **测试分析** | 检查清单 + 表格 | ❌ 可选 | 清单逐项勾选，表格统计覆盖率 |
| **需求分析** | 执行摘要 + 叙事报告 | ❌ 可选 | 摘要给决策者，报告给执行者 |
| **研究调查** | 执行摘要 + 叙事报告 + 表格 + 代码块 | ❌ 可选 | 多格式组合：摘要 + 报告 + 信源评级表 + 证据标注 |
| **部署巡检** | 检查清单 + 状态表 | ❌ 可选 | 逐项检查，通过/失败状态标注 |
| **故障诊断** | 执行摘要 + 时间线叙事 + 代码块 | 推荐 | 摘要给管理层，叙事展示排查过程，代码块展示根因 |
| **性能分析** | 表格 + 可选用图表 | 推荐 | 表格对比指标，Mermaid 图表展示瓶颈 |

## 3. 可视化强制规则

### 3.1 什么是"强制 Mermaid"

当 SKILL 的任务类型被标记为**强制可视化**时，输出内容必须包含至少 1 个 Mermaid 图表。

**强制条件**（满足任一即触发）：

1. 任务涉及**流程/逻辑链**描述（代码审查、业务流程分析）
2. 任务涉及**多组件交互**（架构评审、API 调用分析）
3. 任务涉及**状态转换**（工作流设计、状态机）
4. 任务复杂度为**中等及以上**，且有明确的"变更→影响"关系

**强制可视化规则**：

| 规则 | 说明 |
|:---|:---|
| 最少 1 张图 | 每次输出必须包含至少 1 个 Mermaid 图表 |
| 复杂变更 2 张图 | 重要变更需要业务流 + 技术流两张图 |
| 优先流程图和序列图 | `flowchart` 或 `sequenceDiagram`，展示过程/调用/数据流 |
| 禁止纯分类框图 | 只做分组无流向关系的框图不允许 |
| 强制配色 | 填充色和文字色必须显式指定，确保明暗主题都可读 |
| 图表类型匹配 | 架构→`graph TD`，调用→`sequenceDiagram`，状态→`stateDiagram`，类→`classDiagram` |

### 3.2 可视化豁免条件

以下情况可豁免强制可视化：

- 任务类型为纯文本分析（无流程关系）
- 输出内容为单一结论（如"通过/不通过"）
- SKILL 复杂度为**简单**（单文件、无 references）

### 3.3 在 SKILL 中声明可视化规则

在输出规格区块中使用以下格式声明：

```
## 输出规格

### 强制可视化
- **Mermaid 流程图**：展示审核代码的业务逻辑链（flowchart TD/LR）
- **Mermaid 序列图**（复杂变更）：展示关键函数的调用链（sequenceDiagram）
- **配色要求**：填充色 #eee（浅色）/ #333（深色），文字色 #333（浅色）/ #eee（深色）
```

## 4. 输出结构层次规范

### 4.1 执行摘要（必选，中等及以上复杂度）

紧接标题之后，供决策者快速了解全貌：

```
## 执行摘要

**总体判决**：✅ 通过 | ⚠️ 有条件通过 | 🔶 需修改 | ❌ 驳回

**风险等级**：🟢 低 | 🟡 中 | 🟠 高 | 🔴 极危

**关键发现**：
1. [最重要的发现，一句话]
2. [次重要发现]
3. [第三重要发现]
```

### 4.2 问题分级（必选，审查类 SKILL）

所有问题必须标注严重度：

| 等级 | 标识 | 定义 | 动作 |
|:---|:---|:---|:---|
| **极危** | 🔴 Critical | 安全漏洞、数据丢失风险、破坏性 Bug | 必须立即修复，否则阻断合入 |
| **高危** | 🟠 High | 重大缺陷，影响核心功能 | 合入前必须修复 |
| **中危** | 🟡 Medium | 明显问题，建议修复 | 合入前建议修复 |
| **低危** | 🟢 Low | 轻微建议，可选修复 | 可后续优化 |

### 4.3 判决策略（必选，审查类 SKILL）

根据问题分布自动给出判决：

| 极危 | 高危 | 判决 |
|:---|:---|:---|
| > 0 | 任意 | ❌ 驳回 |
| 0 | > 3 | 🔶 需修改 |
| 0 | 1-3 | ⚠️ 有条件通过 |
| 0 | 0 | ✅ 通过（仅中危/低危记录） |

### 4.4 推荐行动（必选，审查类 SKILL）

基于判决给出具体行动项：

```
## 推荐行动

### 合入前必须（对应极危+高危）
- [ ] [行动项描述]
- [ ] [行动项描述]

### 合入后建议（对应中危）
- [ ] [行动项描述]

### 后续优化（对应低危）
- [ ] [行动项描述]
```

### 4.5 正面观察（推荐，所有审查类 SKILL）

不仅报告问题，也要肯定做得好的地方：

```
## 正面观察

- ✅ [代码组织清晰]
- ✅ [错误处理完善]
- ✅ [测试覆盖充分]
```

## 5. 用户交互模式

### 5.1 四种交互模式

| 模式 | 适用场景 | 示例 |
|:---|:---|:---|
| **一次性报告** | 简单查询、研究调查 | 输出完整报告后结束 |
| **确认-修复-验证循环** | 代码审查、问题诊断 | 输出问题 → 用户选择修复 → 修复后验证 → 确认 |
| **分阶段交付** | 复杂分析 | 先出摘要 → 用户确认 → 出详细报告 |
| **异步定时** | 周期性巡检 | 定时执行 → 自动推送到指定位置/频道 |

### 5.2 交互模式声明格式

在 SKILL 的输出规格中，使用以下格式声明：

```
### 用户交互
- **模式**：确认-修复-验证循环
- **交互工具**：AskUserQuestion（首选），文本选项（回退）
- **循环次数**：最多 3 轮修复-验证
- **终止条件**：用户确认通过 / 全部问题已修复 / 达到最大轮数
```

## 6. 输出文件路径规范

### 6.1 输出路径规则

审查/分析类 SKILL 应当在输出规格中指明文件保存路径：

| SKILL 类型 | 输出路径模板 |
|:---|:---|
| 代码审查 | `docs/reviews/{branch-name}-{date}.md` |
| 安全审计 | `docs/security-audits/{scope}-{date}.md` |
| 架构评审 | `docs/architecture/{component}-review-{date}.md` |
| 研究调查 | `docs/research/{topic}-{date}.md` |
| 部署巡检 | `docs/operations/{env}-check-{date}.md` |

### 6.2 路径声明格式

在输出规格末尾使用：
```
### 输出文件
- **路径**：`docs/reviews/{branch-name}-{date}.md`
- **格式**：Markdown，UTF-8 编码
- **覆盖策略**：不覆盖已有文件，date 精确到日
```

## 7. 组合示例

### 示例：代码审查 SKILL 的输出内容规格

```
## 输出规格

### 强制可视化
- **Mermaid 流程图**：展示变更涉及的业务逻辑链（flowchart TD）
- **Mermaid 序列图**（复杂变更，>10 文件）：展示关键组件的调用时序（sequenceDiagram）
- **配色要求**：`fill:#f0f0f0,color:#333` → 每个节点显式指定

### 输出结构
1. 执行摘要（判决 + 风险等级 + 关键发现）
2. 问题分级清单（Markdown 表格，按严重度降序）
3. 正面观察（至少 1 条）
4. 推荐行动（分"合入前必须"和"合入后建议"）
5. Mermaid 可视化图表

### 问题分级
- 🔴 极危：安全漏洞、数据损失、破坏性 Bug
- 🟠 高危：重大功能缺陷
- 🟡 中危：明显代码坏味
- 🟢 低危：轻微改进建议

### 判决策略
按极危+高危数量自动给出：驳回/需修改/有条件通过/通过

### 用户交互
- **模式**：确认-修复-验证循环
- **交互工具**：AskUserQuestion → 提供 [修复全部极危高危/逐个修复/仅查看不改] 选项
- **循环轮数**：最多 3 轮

### 输出文件
- **路径**：`docs/reviews/{branch-name}-{date}.md`
- **格式**：Markdown
```

## 8. 反模式

| 编号 | 反模式 | 表现 | 修正 |
|:---:|:---|:---|:---|
| 输出反模式1 | 无输出格式定义 | SKILL 只定义了工作流，没有告诉 Agent 输出什么、怎么输出 | 至少定义 1 种格式类型 + 输出结构 |
| 输出反模式2 | 纯文字墙 | 审查结果用大段文字描述，用户找不到重点 | 使用表格 + 分级 + 摘要结构 |
| 输出反模式3 | 缺可视化 | 审查流程变更/SDK 调用等适合流程图的内容无图表 | 按 §3.1 强制条件检查是否需要 Mermaid |
| 输出反模式4 | 缺判决 | 只列问题不给结论，用户不知道要不要合入 | 按 §4.3 加入判决策略 |
| 输出反模式5 | 无正面反馈 | 只报告问题，不肯定做得好的地方 | 加入正面观察区块 |
| 输出反模式6 | 无输出路径 | 输出内容不保存到文件，Agent 下次无法引用 | 指定文件输出路径 |
| 输出反模式7 | 无交互设计 | 一次性输出后没有用户确认和修复闭环 | 根据任务类型选择 §5.1 中的交互模式 |
| 输出反模式8 | 分类框图替代流程 | 用 Mermaid 画了问题分类框图，没有流向关系 | 改用表格展示分类信息，流程图只用于有流向的内容 |

## 9. 设计流程（UR-SKILL 使用本指南）

当 UR-SKILL 生成 SKILL 时，按以下步骤为 SKILL 设计输出内容：

1. **识别任务类型**：代码审查 / 安全审计 / 架构评审 / 研究调查 / 部署巡检 / ...
2. **查格式决策矩阵**：按 §2.2 确定首选格式类型
3. **判断强制可视化**：按 §3.1 检查是否需要 Mermaid；需要则填入强制规则
4. **选择输出结构**：按 §4 确定是否需要执行摘要、问题分级、判决策略、正面观察
5. **设计用户交互**：按 §5.1 选择交互模式，填入交互工具和循环参数
6. **指定输出路径**：按 §6.1 填入输出文件路径模板
7. **写入输出规格**：将以上决策写入 SKILL 的"输出规格"区块
