# 反模式文件设计指南

> 用途：定义反模式文件的标准格式、内容要求和设计方法
> 核心原则：反模式是"看似正确实则有害"的做法，必须解释"为什么错"和"为什么对"

---

## 1. 为什么需要反模式文件

反模式是软件工程标准术语。与"错误"不同，反模式是"好心办坏事"——开发者出于好意图选择了某种做法，但结果有害。

如果不记录反模式：
- LLM 重复生成已被证明有害的做法
- 使用者无法区分"正确做法"和"看似正确实则有害的做法"
- 质量检查无法识别结构性问题（只能发现语法错误）

反模式文件是**教学性参考文件**（L3），按需加载。当 LLM 生成 SKILL 内容时，加载反模式文件进行静态扫描，识别潜在问题。

---

## 2. 反模式的标准格式

### 2.1 六个必要要素

| 要素 | 说明 | 为什么必须 |
|:---|:---|:---|
| **名称** | 简洁、精准的命名 | 便于引用和检索 |
| **表现** | 什么情况下出现，如何识别 | 让使用者能自行诊断 |
| **为什么出现** | 好意图（好心） | 反模式不是错误，是"好心办坏事"；不解释好意图，使用者无法理解为什么会犯这个错 |
| **危害** | 为什么错（办坏事） | 让使用者理解后果，驱动改正 |
| **检测方法** | 如何自动化/手动识别 | 降低检查成本，提高覆盖率 |
| **规避策略** | 替代方案 + 为什么对 | 只说不做X不够，必须说"做Y instead"；必须解释为什么Y是对的 |

> 为什么六个要素：少于六个，无法完整传递"好心→办坏事→为什么→怎么做对"的教学链条。多于六个，信息密度下降，注意力稀释。

### 2.2 格式示例

```markdown
## {编号} {名称}

**表现**：{什么情况下出现，如何识别}

**为什么出现**：{好意图，开发者为什么会选择这个做法}

**危害**：{为什么错，具体后果}

**检测方法**：{如何自动化或手动识别}

**规避策略**：{替代方案} → {为什么对}
```

## 3. 生成方法（ur-skill特殊说明）

> 本 SKILL 是"设计 SKILL 的 SKILL"（ur-skill），自身没有历史项目积累。
> 因此，反模式不是"从历史项目收集"，而是"基于生成的 SKILL 的核心场景，动态调研生成"。

### 3.1 基于联网调查生成

当使用本 SKILL 生成具体 SKILL 时，按以下步骤生成反模式：

1. **确定核心场景**：从用户需求中提取 SKILL 的核心使用场景（如"Python 代码检查""REST API 设计"）
2. **联网调查行业痛点**：搜索该场景下的常见错误、最佳实践反例、Stack Overflow 高频问题
3. **识别反模式**：从调研结果中提取"看似正确实则有害"的做法
4. **验证反模式**：
   - [ ] 不是偶发错误：该做法在多个场景下重复出现
   - [ ] 有好意图：能解释开发者为什么选择这个做法
   - [ ] 有明确危害：能说明具体后果
   - [ ] 有替代方案：能提供正确的替代做法

### 3.2 基于用户指示生成

如果用户提供了具体场景或已有材料：

1. **分析用户材料**：提取用户描述中的常见错误、痛点、失败案例
2. **识别反模式**：将用户描述转化为"表现→危害→规避策略"的反模式结构
3. **补充调研**：对用户未覆盖的场景，联网调查补充
4. **验证完整性**：
   - [ ] 覆盖用户提到的所有痛点
   - [ ] 补充用户未提及但行业常见的反模式
   - [ ] 反模式数量与复杂度匹配（简单 3-4 个，中等 5-6 个，复杂 7-8 个）

### 3.3 ur-skill 自检清单

ur-skill 设计 SKILL 时，必须自检以下反模式，确保生成的 SKILL 不违反 ur-skill 的规范（详细检查流程见 ur-skill.md 工作流校验步骤）：

| 反模式 | 表现 | 规避策略 |
|:---|:---|:---|
| 规范越位 | 将规范定义（字段约束、格式说明）直接塞进 body | 删除规范表格，下沉到 references/ |
| 占位符残留 | 生成的 SKILL body 中存在 [xxx]、TODO、FIXME 等未填充内容 | 全部填充完整，或删除并标注盲区 |
| 示例污染 | 将 references/ 中的示例直接复制填充到 body | 示例通过声明式引用指向 references/ |
| 架构混淆 | 能力矩阵方向是工作流步骤（如"解析 → 分析 → 执行"） | 改为专业领域命名（如"静态分析"） |
| 检查缺失 | 生成的 SKILL 工作流中审视检查数量不足 | 按节点类型分配：关键节点 6 维，非关键节点 3 维 |
| 盲区逃避 | 盲区识别只写"已标注局限"，未按三层机制执行 | 强制按三层机制执行：调查优化 → 请求资源 → 盲区报告 + 可行性建议 |

> 为什么需要ur-skill自身反模式：ur-skill设计 SKILL 时，自身可能违反自己的规范。ur-skill反模式是"自检清单"，确保生成的 SKILL 不继承ur-skill的缺陷。

## 4. 反模式的设计方法

### 4.1 收集反模式

从以下来源收集：
- 历史项目中的重复错误
- 代码审查中的常见问题
- 业界已知的反模式（OWASP, Martin Fowler）
- 本体系自身的违反案例（以身作则）

### 4.2 验证反模式

每个反模式必须通过以下验证：

- [ ] **不是偶发错误**：该做法在多个场景下重复出现
- [ ] **有好意图**：能解释开发者为什么选择这个做法
- [ ] **有明确危害**：能说明具体后果，不是抽象批评
- [ ] **有替代方案**：能提供正确的替代做法
- [ ] **可检测**：能通过规则或脚本自动化识别

### 4.3 组织反模式

按检测优先级排序：

| 优先级 | 排序依据 | 示例 |
|:---|:---|:---|
| 极危 | 直接影响可用性 | 反模式2 占位符残留（交付物不完整） |
| 高危 | 直接影响专业性/安全性 | 反模式1 规范越位 / 反模式7 风险边界滥用 / 反模式9 盲区甩锅 |
| 中危 | 直接影响质量深度 | 反模式3 示例污染 / 反模式4 架构混淆 / 反模式5 检查缺失 / 反模式8 切面填充 |
| 低危 | 直接影响持续优化 | 反模式6 盲区逃避（优化停滞） |

> 为什么按优先级排序：高优先级反模式先检测，先修复。如果 极危 未通过，无需检测 高危-P6。

## 5. 反模式文件的内容要求

### 5.1 必须说明的内容

- **每个反模式必须有"为什么出现"**：不解释好意图，反模式变成教条清单（"不要做X"），使用者不知道X为什么有害。
- **每个反模式必须有"规避策略"**：只说不做X不够，必须说"做Y instead"。
- **每个规避策略必须有"为什么对"**：不解释为什么Y是对的，使用者可能选择另一个错误做法。

### 5.2 禁止的内容

- **禁止只写"不要做X"**：必须有替代方案和理由。
- **禁止抽象批评**：如"这个做法不好"，必须具体说明"为什么不好""有什么后果"。
- **禁止无检测方法**：反模式必须可检测，否则无法执行质量检查。

## 6. 扫描脚本设计

反模式文件可包含扫描脚本，用于自动化检测。

### 6.1 脚本要求

- **只读**：脚本只检测，不修改文件
- **幂等**：多次执行结果一致
- **自包含**：不依赖外部环境
- **输出结构化**：输出格式统一，便于解析

### 6.2 脚本示例

```python
# 占位符残留检测
PLACEHOLDER_PATTERN = r'\[.*?\]|TODO|FIXME|\{.*?\}'
matches = re.findall(PLACEHOLDER_PATTERN, body_content)
if matches:
    report(f"反模式2 占位符残留: {matches}")
```

### 6.3 其他反模式扫描示例

```python
# 反模式5 检查缺失扫描
for step in workflow_steps:
    check_count = step.count("- [ ]")
    if step.is_critical and check_count < 6:
        report(f"反模式5 检查缺失: 关键步骤 {step.name} 只有 {check_count} 个检查项")

# 反模式7 风险边界滥用扫描
CAPABILITY_DEGRADE_KEYWORDS = [
    "不负责", "不属于", "不在.*范围", "不管", "不承担",
    "只做.*不做", "仅限于", "不涉及", "不包含"
]
rb_lines = re.findall(r'\|\s*风险边界-\d+\s*\|\s*(.+?)\s*\|', body_content)
for line in rb_lines:
    for kw in CAPABILITY_DEGRADE_KEYWORDS:
        if re.search(kw, line):
            report(f"反模式7 风险边界滥用: 疑似能力降级 '{kw}' -> '{line.strip()[:60]}'")

SAFETY_KEYWORDS = ["违法", "公序良俗", "歧视", "攻击", "注入", "越狱", "恶意", "漏洞", "爬取", "侵犯"]
has_safety = any(re.search(kw, body_section_of_risk_boundaries) for kw in SAFETY_KEYWORDS)
if not has_safety:
    report(f"反模式7 风险边界滥用: 风险边界声明缺少安全语义，疑似仅为范围描述")

# 反模式8 切面填充扫描
FACET_FILLER_PATTERNS = [
    r"掌握相关领域知识",
    r"识别潜在风险",
    r"保证输出质量",
    r"考虑全局影响",
    r"优化资源投入",
]
for pattern in FACET_FILLER_PATTERNS:
    if re.search(pattern, body_content):
        report(f"反模式8 切面填充: 疑似通用套话 '{pattern}'")

# 反模式9 盲区甩锅扫描
BLIND_DUMP_KEYWORDS = ["仅供参考", "请自行验证", "可能存在.*盲区", "不保证"]
blind_sections = re.findall(r'盲区[：:](.*?)(?=\n\n|\Z)', body_content, re.DOTALL)
for section in blind_sections:
    if any(re.search(kw, section) for kw in BLIND_DUMP_KEYWORDS):
        if "已尝试" not in section:
            report(f"反模式9 盲区甩锅: 纯声明式盲区，无已尝试动作")
```

> 为什么脚本只检测不修改：修改操作风险高，应由人类确认后执行。检测脚本可集成到 CI/CD 流程中，自动化拦截问题。

## 7. 完整性检查清单

设计反模式文件时，逐项检查：

- [ ] 每个反模式包含六个要素（名称、表现、为什么出现、危害、检测方法、规避策略）
- [ ] 每个反模式有"为什么出现"（好意图）
- [ ] 每个反模式有"规避策略"（替代方案 + 为什么对）
- [ ] 反模式按检测优先级排序
- [ ] 反模式不是偶发错误，是重复出现的模式
- [ ] 扫描脚本只读、幂等、自包含
- [ ] 文件 < 200 行，超过则拆分
- [ ] 无占位符残留（自身不能违反反模式）
