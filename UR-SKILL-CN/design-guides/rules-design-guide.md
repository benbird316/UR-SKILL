# 规则设计指南

> 用途：定义生成 SKILL 的规则体系（MUST / MUST NOT / SHOULD / SHOULD NOT / MAY）设计原则
> 核心原则：规则是可验证的行为约束；门控是过程检查，反模式是结果检查，三者互补不替代

---

## 1. 规则设计原则

1. 规则必须是该 SKILL 领域内可验证的行为约束，不是对目标平台的假设或对用户的猜测
2. 禁止抽象规则（如"做好工作"、"确保质量"）
3. 禁止自相矛盾：不同规则之间必须逻辑一致，不能同时出现 MUST A 和 MUST NOT A
4. MUST NOT 不重复边界声明的具体内容，声明是边界的权威来源

---

## 2. RFC 2119 关键词语义

规则模板遵循 RFC 2119（及 RFC 8174）标准，关键词在**全大写**时具有特殊含义，可被工具机械解析：

| 关键词 | 含义 |
|:---|:---|
| MUST | 强制要求 |
| MUST NOT | 强制禁止 |
| SHOULD | 强烈推荐 |
| SHOULD NOT | 强烈不推荐 |
| MAY | 可选 |

---

## 3. 门控与反模式的关系

| 机制 | 类型 | 执行时机 | 作用 |
|:---|:---|:---|:---|
| 门控 | 过程检查 | 动态，每步执行 | 实时校验规则遵守情况 |
| 反模式扫描 | 结果检查 | 静态，交付前一次 | 识别"看似正确实则有害"的做法 |

两者互补，不替代。门控检查规则执行过程，反模式扫描检查最终交付物。

---

## 4. 规则、门控、反模式、风险边界、专业边界职责分离

| 元素 | 职责 | 禁止 |
|:---|:---|:---|
| 规则 | 可验证的行为约束 | 抽象、矛盾、重复边界声明 |
| 门控 | 过程检查点，引用规则编号 | 重复规则内容 |
| 反模式 | 结果检查，识别有害做法 | 包含通用反模式（由 UR-SKILL 统一处理） |
| 风险边界 | 安全红线 | 含范围限制或能力降级 |
| 专业边界 | 越界防护 | 含安全声明或能力降级 |

五者职责分离，不重叠。
