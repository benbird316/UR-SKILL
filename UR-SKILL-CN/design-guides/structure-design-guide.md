# 结构规范指南

> 只教怎么组织 SKILL.md 及配套文件的格式符号、信息组织方式、注意力对抗策略。
> 判定结构规范需求，见 skill-package-design-guide.md §2。

---

## §1 通用格式符号

### 1.1 约束等级符号（RFC 2119）

| 符号 | 含义 | 偏离处理 |
|:---|:---|:---|
| **MUST** | 绝对要求 | 违反 = 不兼容规范 |
| **MUST NOT** | 绝对禁止 | 违反 = 不兼容规范 |
| **SHOULD** | 强烈建议 | 偏离需记录理由 |
| **SHOULD NOT** | 不推荐 | 偏离需记录理由 |
| **MAY** | 真正可选 | 无偏离概念 |

### 1.2 检查符号

| 符号 | 含义 | 使用场景 |
|:---|:---|:---|
| `- [ ]` | 未检查 | 检查清单、审视项 |
| `- [x]` | 已检查 | 检查清单完成标记 |
| `1. 2. 3.` | 步骤编号 | 工作流序列 |
| `规则01` | 规则编号 | 规则引用 |
| `反模式1` | 反模式编号 | 反模式引用 |
| `风险边界-01` | 风险边界编号 | 风险边界声明引用 |

### 1.3 强调符号

| 符号 | 含义 | 使用场景 |
|:---|:---|:---|
| `**加粗**` | 关键约束 | 规则关键词、检查项核心词 |
| `> 引用块` | 核心原则 | 需要强调的原则性内容 |
| `` `行内代码` `` | 格式示例 | 文件路径、配置项 |
| ` ``` ` 代码块 | 完整示例 | YAML frontmatter、配置模板 |
| `---` | 模块分隔 | YAML frontmatter 分隔、章节分隔 |

### 1.4 不使用的符号

- **不用 emoji** 作为约束标记（规则关键词、检查项核心词、门控条件中不用 emoji），无法被所有终端解析
- **SHOULD 按需使用 emoji** 在输出模板/报告示例中（如问题分级 🔴🟠🟡🟢），提升视觉区分度，但不得在约束规则中使用
- **不用 Unicode 特殊符号**（如 ◈）：编码风险，无业界对应
- **不用 ASCII 艺术表格线**（├ ┌ ┬）：Markdown 标准表格 `|` 足够

## §2 信息组织方式

### 2.1 一维信息用列表

- 规则声明：bullet list，每条独立，以 RFC 2119 关键词开头
- 步骤序列：编号列表 `1. → 2. → 3.`
- 检查清单：`- [ ]` 复选框

### 2.2 二维信息用表格

二维关系（对比、映射、预算分配）用标准 Markdown 表格 `| col1 | col2 | col3 |`。

### 2.3 层次递进

- **L1 总纲**：段首 1-2 句，说明文件用途和核心原则
- **L2 模块**：`##` 二级标题，划分独立主题
- **L3 条目**：列表或表格，具体规范条目
- **L4 细节**：移入 `references/`，不在 body 中展开

---

## §3 正向表述

| 反例（反向/抽象） | 正例（正向/具体） |
|:---|:---|
| 不得丢失原文语义 | **MUST** 保留原文语义 |
| 禁止生成可执行代码 | **MUST NOT** 在 body 中嵌入可执行代码 |
| 不能把示例直接填充 | **MUST NOT** 将示例直接复制到 body |
| 不得出现内部术语 | **MUST NOT** 在 body 中使用内部术语 |

## §4 注意力对抗策略

### 4.1 位置策略（对抗 Lost in the Middle）

- 文件首段：放核心能力摘要 + 关键约束
- 文件尾段：放检查清单 + 关键限制 + 版本信息
- 避免：关键规则放在 body 中间段

### 4.2 重复策略（对抗 Attention Dilution）

- 核心规则在规则章节声明，在工作流中引用编号
- 关键输出格式在规范中定义，在示例中展示
- 不同表述，避免 verbatim 重复

### 4.3 信号强化

- 关键约束用 `**加粗**`
- 核心原则用 `> 引用块`
- 格式示例用 ` ``` ` 代码块
- 规则 / 门控 / 反模式用编号系统

### 4.4 Body 区块推荐排列顺序（双重锚定策略）

| 区块 | 位置 | 放什么 | 原因 |
|:---|:---|:---|:---|
| **首因区** | body 前 20% | 角色声明、任务定义、核心约束（MUST/MUST NOT） | 注意力最高，决定行为基调 |
| **中间区** | body 中间 60% | 能力矩阵、工作流步骤 | Lost in the Middle 风险区，放索引和引用路径 |
| **近因区** | body 后 20% | 输出格式契约、关键规则摘要、风险边界声明、参考引用路径 | 次高注意力，核心规则重复增加捕获概率 |

**Double Prompting 策略**：在近因区用**不同表述重复**首因区的核心约束（不写 verbatim 副本）。

## §5 信息密度

保持高信息密度：删除模式化填充、避免冗余声明、每个句子传递独立信息。

**检测方法**：
- 删除"众所周知""需要注意的是"等模式化前缀
- 删除重复声明（如规则在工作流中重复写出内容，应引用编号）
- 删除无信息增量的过渡句
- 检查是否每个段落都有独立信息价值

**不适用数值阈值**：信息密度是设计原则，用检查方法替代不精确的数值阈值。

## §6 检查清单

- [ ] 规则使用 RFC 2119 关键词（MUST / MUST NOT / SHOULD / SHOULD NOT / MAY）
- [ ] 一维信息用列表，二维信息用表格
- [ ] 层次递进：L1 总纲 → L2 模块 → L3 条目 → L4 细节
- [ ] 无自创符号（emoji、Unicode 特殊符号、ASCII 艺术表格线）
- [ ] 正向表述：MUST / SHOULD / MAY 替代 "不得/禁止/必须"
- [ ] 关键信息在首尾段重复或引用
- [ ] 无抽象规则（如"做好工作"）
- [ ] 无矛盾规则（如 规则01 说 MUST A，规则02 说 MUST NOT A）
- [ ] 规则、门控、反模式、风险边界四者职责分离，不重叠
- [ ] 门控检查项引用规则编号，不重复规则内容
