# 方法类 ref 设计指南

> 只教怎么写方法类 ref 文件。方法类 ref 回答"怎么找？"——定位方法 + 追溯策略 + 工具。

## 格式

```markdown
> 加载阶段：步骤 2

## {方法名称}

**适用场景**：{什么情况用这个方法}

**前置条件**：{执行前必须满足的条件——需要的权限、环境、数据}

**输入**：
| 参数 | 类型 | 必填 | 说明 |
|:---|:---|:---|:---|
| {参数名} | {类型} | 是/否 | {一句话} |

**步骤**：
1. [{工具类别}] {操作} → {预期输出}
2. [{工具类别}] {操作} → {预期输出}

**返回值**：
| 输出 | 类型 | 说明 |
|:---|:---|:---|

**降级路径**：{首选工具不可用时，备选方案}

**常见错误**：
| 错误 | 原因 | 处理 |
|:---|:---|:---|
```

## 规则

- 每个步骤必须绑定工具类别（`[文本搜索]`、`[命令执行]`、`[文件读取]` 等）
- 每个操作必须有预期输出——不是"执行 X"，是"执行 X → 应符合 Y"
- 必须包含降级路径——工具的备选方案或人工替代
- 不写分类定义（那是说明类的事）、不写修复策略（那是方案类的事）
- 必须定义**输入签名**（参数表的类型+必填列不可省略）和**返回值表**，遵循 ref-types-design-guide.md §1.2 可引用性原则
- 必须列出**常见错误**及处理方法——方法在没有正确/错误反馈时无法自我校准
- 必须标注使用的方法/API 的**来源**（K类 + 信源 + 获取日期），遵循时效性原则

## 长度

≤ 150 行。每个方法一般 10-20 行，4-7 个方法为宜。
