# 知识型参考文件设计指南

> 用途：定义知识型参考文件的标准格式、子类型分类与设计方法
> 核心原则：知识型文件是"知道什么"，不是"怎么做"或"什么不能做"——与操作型、教学型、规范型文件职责分离

---

## 1. 为什么需要知识型参考文件设计指南

当 UR-SKILL 仅有 5 个设计指南时，它们覆盖了操作型（反模式、故障诊断）、教学型（示例）、规范型（规范定义）、资源型（模板/Schema），但缺失了最基础的类型：**知识型文件**。本指南填补了这个空缺。

知识型文件存储 SKILL 在执行任务时需要的**领域专业知识**——不是"怎么做"（工作流），不是"什么不能做"（反模式），而是"这个领域的事实和规则是什么"。

如果不定义知识型文件：
- 领域知识被塞进 SKILL.md body → body 膨胀，违反渐进式加载
- 领域知识被分散在多个文件中 → 重复、不一致、无法维护
- LLM 需要依赖训练数据中的通用知识 → 无法获取 SKILL 专属的领域深度
- 知识更新需要修改多个文件 → 变更风险高

**典型场景**：
- Python 代码安全审计 SKILL 需要 OWASP Python Top 10 漏洞详解 → 知识型文件
- REST API 设计 SKILL 需要 HTTP 状态码参考与 RESTful 约束 → 知识型文件
- 前端组件开发 SKILL 需要 React Hooks 规则与性能模式 → 知识型文件
- 金融分析 SKILL 需要估值模型公式与市场规则 → 知识型文件

知识型文件是**知识型参考文件**（L3），按需加载。当 LLM 需要特定领域的专业知识来完成工作流步骤时，加载知识型文件。

## 2. 知识型文件的子类型分类

### 2.1 四大子类型

| 子类型 | 说明 | 典型文件名 | 何时使用 |
|:---|:---|:---|:---|
| **K1 领域知识** | 专业领域的概念、规则、方法论 | `domain-knowledge.md`, `owasp-top10.md` | SKILL 的核心任务依赖特定领域的专业知识 |
| **K2 API/CLI 参考** | 接口参数、命令语法、返回值 | `api-reference.md`, `cli-reference.md` | SKILL 需要调用特定的外部工具或 API |
| **K3 配置/策略文档** | 配置 Schema、公司政策、业务规则 | `policy-rules.md`, `config-schema.md` | SKILL 需要遵循组织特定的规则或配置 |
| **K4 设计模式** | 可复用的代码/架构模式 | `patterns.md`, `architecture-patterns.md` | SKILL 需要在多种场景下应用同一组模式 |

> 为什么分子类型：不同类型的知识型文件有不同的要素需求。API 参考需要参数表格，领域知识需要概念说明，模式指南需要场景→方案→权衡。统一格式会丢失子类型特有的信息结构。

### 2.2 子类型选择决策树

```
SKILL 需要知识型参考文件？
├── 需要调用外部工具/API？ → K2 API/CLI 参考
├── 需要遵循组织规则/配置？ → K3 配置/策略文档
├── 需要应用可复用模式？ → K4 设计模式
├── 其他领域专业知识？ → K1 领域知识
└── 以上皆非？ → 可能不需要知识型文件
```

## 3. 各子类型的标准格式

### 3.1 K1 领域知识

**三要素**：

| 要素 | 说明 | 为什么必须 |
|:---|:---|:---|
| **概念定义** | 精确定义核心概念 | 知识的入口；不定义概念，后续规则与约束无锚点 |
| **规则/约束** | 该领域的核心规则 | 知识的可操作性；只有定义没有规则 = 百科全书而非工作知识 |
| **优先级/权衡** | 规则冲突时的优先级 | 现实场景中规则常冲突；不说明优先级，LLM 无法决策 |

**格式模板**：

```markdown
## {概念名称}

**定义**：{一句话精确定义}

**规则**：
- {规则 1}
- {规则 2}
- {规则 3}

**优先级**：当 {规则 A} 与 {规则 B} 冲突时 → {决策规则}
```

**示例（OWASP Python Top 10 片段）**：

```markdown
## SQL 注入（SQL Injection）

**定义**：攻击者通过用户输入拼接 SQL 语句，执行非预期的数据库操作。

**规则**：
- MUST 使用参数化查询（parameterized query）而非字符串拼接
- MUST NOT 将用户输入直接嵌入 SQL 语句
- SHOULD 使用 ORM 而非原生 SQL（除非性能要求）

**优先级**：ORM 便利性 vs 参数化查询 → 参数化查询优先。ORM 生成的 SQL 仍需审查。
```

### 3.2 K2 API/CLI 参考

**三要素**：

| 要素 | 说明 | 为什么必须 |
|:---|:---|:---|
| **调用签名** | 命令/方法的基本签名 | 知识入口；不知道签名就无法调用 |
| **参数表** | 参数名、类型、必填、默认值、说明 | 知识核心；参数是调用正确性的关键 |
| **返回值/输出** | 正常输出、错误输出格式 | 知识边界；不知道返回值格式无法处理结果 |

**格式模板**：

```markdown
## {工具/API 名称}

### 调用签名
`{command_signature}` 或 `{method_signature}`

### 参数

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|:---|:---|:---:|:---|:---|
| {param1} | {type} | MUST | {default} | {说明} |

### 返回值
- 成功：{格式与示例}
- 失败：{错误格式与常见错误码}
```

### 3.3 K3 配置/策略文档

**三要素**：

| 要素 | 说明 | 为什么必须 |
|:---|:---|:---|
| **策略声明** | 规则的内容 | 知识主体 |
| **适用范围** | 规则何时适用、何时不适用 | 边界不清 = 规则滥用或漏用 |
| **违反处理** | 违反时的后果 | 赋予规则执行力 |

**格式模板**：

```markdown
## {策略名称}

**规则**：{策略内容}

**适用范围**：
- 适用：{场景 A, 场景 B}
- 不适用：{场景 C}

**违反处理**：{违反时的具体动作}
```

### 3.4 K4 设计模式

**四要素**：

| 要素 | 说明 | 为什么必须 |
|:---|:---|:---|
| **场景** | 什么情况下使用 | 模式匹配入口 |
| **方案** | 具体怎么做 | 模式内容 |
| **权衡** | 方案的利弊 | 模式选择依据 |
| **不适用场景** | 什么情况下不要用 | 反模式预防 |

**格式模板**：

```markdown
## {模式名称}

**场景**：{在什么需求/约束下使用}

**方案**：{具体做法，附代码/配置示例}

**权衡**：
- 优势：{好处}
- 代价：{代价}

**不适用场景**：{什么情况下不要用，为什么}
```

## 4. 知识型文件的通用设计原则

### 4.1 原子性

一个文件一个主题。K1 领域知识按域拆分、K2 API 参考按模块拆分、K3 策略按规则组拆分、K4 模式按模式域拆分。

> 为什么原子性：渐进式加载的核心——只加载需要的知识。一个文件包含 SQL 注入 + XSS + CSRF → LLM 只需要 SQL 注入知识时，被迫加载全部。

### 4.2 可引用性

知识型文件中的每条规则/约束应有编号，便于 SKILL.md body 引用。

```markdown
## K1.1 参数化查询

**规则**：
- {KN-01} MUST 使用参数化查询 ...
```

### 4.3 时效性标注

知识型文件中的每条知识应有来源和时效性标注。

```markdown
> **来源**：OWASP Top 10 2021 | **时效**：长期有效 | **审查周期**：12 个月
```

### 4.4 与 body 的关系

body 不复制知识型文件的内容。body 通过引用触发加载：

```markdown
**K1 领域知识**：审计 Python 代码时，按需加载 references/owasp-python-top10.md
```

## 5. 生成方法（ur-skill 特殊说明）

> 本 SKILL 是"设计 SKILL 的 SKILL"（ur-skill），自身不直接使用知识型文件（ur-skill 自己的领域知识已分布在 design-guides/ 和 templates/ 中）；但为生成的 SKILL 提供知识型文件设计能力。

### 5.1 为生成的 SKILL 创建知识型文件

1. **从能力域推导**：每个辐射领域可能需要对应的知识型文件
   - 如"安全审计"辐射领域 → K1 领域知识（漏洞类型）、K4 设计模式（安全模式）
2. **从工作流步骤判断**：调研步骤中检索到的专业知识 → K1 领域知识；执行步骤中调用的外部工具 → K2 API/CLI 参考
3. **从复杂度判定**：简单 SKILL 不需要独立知识型文件（在 body 中内联）；中等及以上需要
4. **控制知识范围**：只包含 SKILL 执行任务时需要的知识，不追求"该领域的完整知识库"

### 5.2 知识型文件的触发条件

知识型文件是否需要，不由复杂度单独决定，而由 design-rationale.md §9 文件依赖决策树的第 1 步判定：

> **"核心领域是否依赖外部知识库（如 OWASP/RFC/行业标准/法规/API 文档/学术理论体系）？"**
> — 是 → 需要 knowledge-reference 文件（触发即强制），子类型按 §2.2 子类型决策树选择

复杂度与知识型文件的参考关系（仅供参考，最终由决策树判定）：

| 复杂度 | 知识型文件 | 理由 |
|:---|:---|:---|
| 简单 | 不需要（领域知识在 body 中内联） | 简单 SKILL 的知识量少，内联更高效 |
| 中等 | 可能（取决于是否依赖外部知识库） | 依赖外部知识库 → 1-2 个 K1 领域知识文件；不依赖 → 不需要 |
| 复杂 | 通常需要（取决于是否依赖外部知识库） | 依赖外部知识库 → 2-4 个文件（可混合 K1-K4 子类型）；不依赖 → 不需要 |

## 6. 知识型文件与其他参考文件类型的关系

| 对比维度 | 知识型（K） | 操作型（反模式/故障诊断） | 教学型（示例） | 规范型（spec） |
|:---|:---|:---|:---|:---|
| 核心问题 | "知道什么" | "怎么做/什么不能做" | "看这个学" | "标准是什么" |
| 内容性质 | 事实、规则、概念 | 过程、步骤、检测 | Input/Output 对 | 字段约束、验证规则 |
| 更新频率 | 按领域变化 | 按实践积累 | 按场景扩展 | 按标准演进 |
| 典型文件 | owasp-top10.md, react-patterns.md | anti-patterns.md, troubleshooting.md | examples.md | metadata-spec.md |

## 7. 完整性检查清单

设计知识型文件时，逐项检查：

- [ ] 子类型已选（K1/K2/K3/K4 至少一个）
- [ ] 符合子类型的要素要求（K1: 概念定义+规则+优先级; K2: 签名+参数表+返回值; K3: 策略+范围+违反处理; K4: 场景+方案+权衡+不适用场景）
- [ ] 一个文件一个主题（原子性）
- [ ] 每条规则/约束有编号（可引用性）
- [ ] 知识有来源和时效性标注
- [ ] body 不复制知识型文件内容（只引用）
- [ ] 知识范围只覆盖 SKILL 执行所需（不追求完整百科）
- [ ] 文件 < 200 行，超过则拆分为多个原子文件
- [ ] 文件之间不相互引用（渐进式加载约束）
