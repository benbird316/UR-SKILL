# 故障诊断文件设计指南

> 用途：定义故障诊断参考文件（troubleshooting.md）的标准格式、内容要求与设计方法
> 核心原则：每个故障必须配对"症状 → 根因 → 修复动作"，禁止纯声明式提示

---

## 1. 为什么需要故障诊断文件

故障诊断文件记录执行过程中反复出现的失败模式。与反模式不同：
- **反模式**是"设计时做错了什么"——预防性
- **故障诊断**是"运行时坏了怎么办"——修复性

如果不记录故障诊断：
- 同一故障被反复遇到，每次从零诊断，消耗 token 和时间
- 故障修复方法不一致——不同 LLM session 可能给出不同（甚至冲突的）修复动作
- 团队知识无法沉淀——老成员踩过的坑，新 SKILL 继续踩

故障诊断文件是**操作型参考文件**（L3），按需加载。当 LLM 在执行工作流时遇到异常，加载故障诊断文件匹配症状→获取修复动作。

## 2. 故障诊断的标准格式

### 2.1 三个必要要素

| 要素 | 说明 | 为什么必须 |
|:---|:---|:---|
| **症状** | 用户/LLM 能观察到的具体表现 | 诊断的入口；症状描述不精确，LLM 匹配不到正确的故障条目 |
| **根因** | 导致该症状的底层原因 | 只修症状不修根因→下次还坏；理解的修复才可持续 |
| **修复动作** | 具体、可执行的操作步骤 | 知道"为什么坏"但不知道"怎么修"= 无故障诊断价值 |

> 为什么三个要素：症状→根因→修复动作是故障诊断领域（ITIL、Google SRE、DNS Troubleshooting Playbook）的标准三元组。少于三个，诊断链断裂；多于三个，注意力稀释。

### 2.2 格式模板

```markdown
### {编号} {故障名称}

- **症状**：{用户/LLM 能观察到的具体表现，用具体的错误信息、行为异常描述}
- **根因**：{导致症状的底层原因，解释"为什么会这样"}
- **修复动作**：
  1. {具体步骤 1}
  2. {具体步骤 2}
  3. {具体步骤 3}
```

### 2.3 按域分组

故障按**故障域**分组（与工作流步骤或能力域对应），便于 LLM 按上下文快速定位。

```
## 1. 能力矩阵相关故障
## 2. 工作流相关故障
## 3. 内容与格式相关故障
## 4. 质量与安全相关故障
```

> 为什么按故障域分组：LLM 在执行某个工作流步骤时遇到异常，天然知道去"工作流相关故障"域查找，而非遍历全部条目。按域分组比按优先级排序更适合"按需查询"的使用场景。

### 2.4 编号系统

每个故障条目分配 **T{序号}** 编号（T = Troubleshooting），便于跨文件引用。

```markdown
### T01 能力矩阵写成工作流步骤
```

> 为什么编号：与反模式编号（反模式1）、规则编号（规则01）一致，遵循 UR-SKILL 编号系统。SKILL.md body 中可直接引用编号而不重复故障内容。

## 3. 故障诊断的内容要求

### 3.1 必须说明的内容

- **症状必须具体可识别**：不写"输出不对"而写"辐射领域方向为'解析需求 → 检索资料 → 评估来源'"。
- **根因必须解释"为什么"**：不写"设计错误"而写"把执行顺序误当成独立能力域"。
- **修复动作必须可执行**：不写"优化设计"而写"执行排序测试；若逻辑崩坏，退回调研重新推导"。

### 3.2 禁止的内容

- **禁止纯声明式提示**：如"这个问题需要注意"——无根因、无修复动作。
- **禁止无根因的修复**：如只写"重启解决"——不知道为什么坏，下次遇到变体故障无法泛化。
- **禁止不可执行的修复**：如"改进设计""提高质量"——动词太抽象，LLM 无法翻译为具体操作。
- **禁止无编号**：故障条目必须编号，否则无法被 SKILL.md body 引用。

### 3.3 症状描述质量标准

| 标准 | 说明 | 反例 → 正例 |
|:---|:---|:---|
| **可观察** | LLM 能在执行中自主识别 | "能力矩阵设计不合理" → "辐射领域是'解析需求 → 检索资料 → 生成报告'" |
| **具体** | 包含具体的错误形式、数值、输出样本 | "格式不对" → "body 行数超过 500 行" |
| **区分性** | 能区分不同故障条目 | 两个故障条目的症状描述不应让 LLM 无法判断匹配哪个 |

## 4. 生成方法（ur-skill 特殊说明）

> 本 SKILL 是"设计 SKILL 的 SKILL"（ur-skill），自身需要故障诊断文件；同时也为生成的 SKILL 提供故障诊断设计能力。

### 4.1 为 ur-skill 自身创建故障诊断

ur-skill 的故障来源：
- 执行 UR-SKILL 工作流时的常见失败模式（实践体验）
- 生成的 SKILL 被校验步骤拦截时的典型问题
- 反模式检测出的高频违反（从反模式到故障的映射）

收集方法：
1. **从反模式映射**：每个反模式都可能触发故障。如"反模式4 架构混淆"→"T01 能力矩阵写成工作流步骤"
2. **从校验失败记录**：校验步骤（第 5 步）中最常被拦截的问题
3. **从用户反馈**：生成的 SKILL 用户报告的问题

### 4.2 为生成的 SKILL 创建故障诊断

当生成的 SKILL 包含 `references/troubleshooting.md` 时：

1. **从反模式逆推**：每个反模式都可能在运行时表现为故障 → 为高危反模式创建对应的故障条目
2. **从工作流步骤预见**：每个工作流步骤可能失败→为每个步骤创建至少 1 个故障条目
3. **联网调查补充**：搜索该领域的常见失败模式
4. **控制数量**：8-15 个故障条目，按 3-5 个故障域分组

### 4.3 复杂度与故障诊断（故障诊断是 3 底线之一，中等+复杂度强制要求，见 design-rationale.md §9）

| 复杂度 | 故障诊断文件 | 理由 |
|:---|:---|:---|
| 简单 | 不需要 | 简单 SKILL 的故障模式有限，可内联在 body 中 |
| 中等 | 需要，8-12 个故障条目 | 中等 SKILL 有独立工作流，故障需独立记录 |
| 复杂 | 需要，12-15 个故障条目 | 复杂 SKILL 的工作流步骤多，故障域多 |

## 5. 故障诊断与反模式的关系

故障诊断与反模式是不同的参考文件类型，但有映射关系：

| 维度 | 反模式（anti-patterns.md） | 故障诊断（troubleshooting.md） |
|:---|:---|:---|
| 阶段 | 设计时（预防） | 运行时（修复） |
| 触发 | 校验步骤的静态扫描 | 执行过程中的异常匹配 |
| 关注点 | "怎么做是错的" | "坏了怎么修" |
| 因果方向 | 做法 → 危害 | 症状 → 根因 → 修复 |

**映射关系**：一个反模式可能对应零个或多个故障条目。不是每个反模式都需要故障条目——只有那些必然导致可观察症状的反模式才需要。

| 反模式 | 对应故障条目 | 映射理由 |
|:---|:---|:---|
| 反模式4 架构混淆 | T01 能力矩阵写成工作流步骤 | 架构混淆导致能力矩阵设计错误——可观察的症状 |
| 反模式2 占位符残留 | T(占位符扫描失败) | 占位符残留导致静态扫描报错——可观察的症状 |
| 反模式6 盲区逃避 | T(盲区声明无效) | 盲区逃避导致检查清单失败——可观察的症状 |

## 6. 完整性检查清单

设计故障诊断文件时，逐项检查：

- [ ] 每个故障条目有三个要素（症状、根因、修复动作）
- [ ] 症状具体可观察，LLM 能自主识别
- [ ] 根因解释"为什么"，而非仅描述"哪里错了"
- [ ] 修复动作可执行，有具体步骤序列
- [ ] 按故障域分组（3-5 个域）
- [ ] 故障条目有 T{序号} 编号
- [ ] 故障数量与复杂度匹配（中等 8-12，复杂 12-15）
- [ ] 无纯声明式提示（如"需要优化"）
- [ ] 无不可执行的修复（如"改进设计"）
- [ ] 文件 < 200 行，超过则拆分
