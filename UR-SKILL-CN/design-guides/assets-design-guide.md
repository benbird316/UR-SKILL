# 资源型文件设计指南

> 用途：定义 assets/ 目录下静态资源（模板、schema、品牌素材）的设计方法
> 核心原则：资源型文件是可复制、可填充、版本一致的静态内容

---

## 1. 为什么需要资源型文件设计指南

assets/ 目录存放静态资源：
- 输出模板（JSON schema、XML 模板、Markdown 模板）
- 品牌素材（logo、配色方案、字体规范）
- 配置文件（YAML 配置、JSON 配置、环境变量模板）

与脚本不同，资源型文件：
- **不可执行**：没有逻辑，只有数据
- **可填充**：有占位符，运行时填充
- **可复制**：可复用于多个 SKILL

如果不定义资源型文件：
- 输出格式不一致
- 品牌风格混乱
- 配置重复定义

资源型文件是**资源型参考文件**（L3），按需加载。当生成 SKILL 输出时，加载资源型文件填充内容。

---

## 2. 资源型文件的标准格式

### 2.1 三个必要要素

| 要素 | 说明 | 为什么必须 |
|:---|:---|:---|
| **模板结构** | 文件的整体结构 | 定义输出格式，确保一致性 |
| **占位符定义** | 可填充的变量/占位符 | 定义哪些部分可动态填充 |
| **填充规则** | 占位符的填充规则 | 定义填充时的约束和验证 |

> 为什么三个要素：模板结构定义"长什么样"，占位符定义"哪些可变"，填充规则定义"怎么变"。少于三个，无法完整定义资源；多于三个，信息密度下降。

### 2.2 格式示例

```markdown
# {资源名称} 模板

> 用途：{一句话说明用途}
> 版本：{版本号}
> 最后更新：{日期}

---

## 模板结构

```json
{
  "name": "{{name}}",
  "description": "{{description}}",
  "type": "{{type}}",
  "metadata": {
    "updated": "{{updated}}"
  }
}
```

## 占位符定义

| 占位符 | 类型 | 约束 | 必填 |
|:---|:---|:---|:---:|
| `{{name}}` | string | kebab-case, <= 64 字符 | MUST |
| `{{description}}` | string | <= 1024 字符, "Use when..." 开头 | MUST |
| `{{type}}` | string | 固定 "prompt" | MUST |
| `{{updated}}` | string | YYYY-MM-DD 格式 | MUST |

## 填充规则

- **MUST** 填充所有 MUST 占位符，未填充则输出无效
- **MUST** 验证填充值符合约束（如 name 符合 kebab-case）
- **MAY** 填充可选占位符，未填充则保留默认值
- **MUST NOT** 填充非法值（如 name 包含大写字母）
```

## 3. 生成方法（ur-skill特殊说明）

> 本 SKILL 是"设计 SKILL 的 SKILL"（ur-skill），自身没有历史项目积累。
> 因此，资源型文件不是"从历史模板收集"，而是"基于生成的 SKILL 的核心场景，动态设计"。

### 3.1 基于场景需求生成

当使用本 SKILL 生成具体 SKILL 时，按以下步骤生成资源型文件：

1. **确定资源需求**：从用户需求中提取是否需要模板、schema、配置文件
   - 简单 SKILL：通常不需要资源型文件（输出格式简单，无需模板）
   - 中等 SKILL：可能需要 1-2 个模板（JSON schema、Markdown 模板）
   - 复杂 SKILL：可能需要 3-5 个资源（多种输出格式、品牌素材、配置文件）

2. **设计模板结构**：
   - 确定输出格式（JSON / XML / YAML / Markdown）
   - 设计模板整体结构（字段、嵌套、数组）
   - 定义占位符（哪些部分可动态填充）

3. **定义填充规则**：
   - 必填占位符：未填充则输出无效
   - 可选占位符：未填充则保留默认值
   - 验证规则：填充值必须符合约束（如 kebab-case、<= 64 字符）

4. **验证资源型文件**：
   - [ ] 模板结构可验证（JSON Schema / XML Schema / YAML Lint）
   - [ ] 所有占位符有定义（名称、类型、约束、必填）
   - [ ] 占位符命名符合规范（双大括号、kebab-case、语义化）
   - [ ] 填充规则明确（必填/可选、验证方法、默认值）

### 3.2 基于用户指示生成

如果用户提供了具体模板或已有资源：

1. **分析用户资源**：提取用户描述中的模板结构、占位符、填充规则
2. **设计资源型文件**：将用户描述转化为"模板结构 + 占位符定义 + 填充规则"结构
3. **补充设计**：对用户未覆盖的占位符，基于场景补充
4. **验证完整性**：
   - [ ] 覆盖用户提到的所有占位符
   - [ ] 补充用户未提及但场景需要的占位符
   - [ ] 模板有版本号，便于追踪变更

### 3.3 ur-skill 资源模板

ur-skill 自身使用的资源型文件如下（这些文件是 ur-skill 的参考文件，不是生成的 SKILL 的参考文件）：

| 资源文件 | 用途 | 复杂度 |
|:---|:---|:---:|
| ../templates/output-template.md 中的 YAML frontmatter 模板 | 生成 SKILL 的 frontmatter 结构 | 所有复杂度 |
| ../templates/capability-architecture-template.md 中的能力矩阵表格模板 | 生成能力矩阵的标准格式 | 所有复杂度 |
| ../templates/workflow-template.md 中的工作流步骤模板 | 生成工作流的标准格式 | 所有复杂度 |
| ../templates/rules-template.md 中的规则列表模板 | 生成规则的标准格式 | 所有复杂度 |
| JSON schema（验证生成 SKILL 的 frontmatter） | 自动化验证 frontmatter 格式 | 复杂 |

> 为什么ur-skill需要资源型文件：ur-skill设计 SKILL 时，需要统一的模板结构。这些资源型文件是ur-skill的"模板库"，确保所有生成的 SKILL 结构一致。

## 4. 资源型文件的设计方法

### 4.1 模板设计

模板必须是**可验证**的：

| 验证项 | 方法 | 为什么 |
|:---|:---|:---|
| **结构验证** | JSON Schema / XML Schema / YAML Lint | 确保模板本身格式正确 |
| **占位符验证** | 正则扫描 `{{.*?}}` | 确保所有占位符有定义 |
| **填充验证** | 填充后验证输出格式 | 确保填充后输出有效 |
| **版本验证** | 版本号匹配 | 确保使用正确版本的模板 |

### 4.2 占位符设计

占位符命名规范：

| 命名规范 | 说明 | 示例 |
|:---|:---|:---|
| **双大括号** | `{{placeholder}}` | 与 Mustache / Handlebars 模板引擎兼容 |
| **kebab-case** | 小写 + 连字符 | `{{output-format}}` |
| **语义化** | 名称反映内容 | `{{name}}` 不是 `{{n}}` |
| **唯一性** | 不重复定义 | 同一模板中不定义两个 `{{name}}` |

> 为什么双大括号：Mustache / Handlebars 是业界标准模板引擎，双大括号 `{{}}` 是标准占位符语法。单大括号 `{}` 可能与 JSON 对象混淆。

### 4.3 版本控制

资源型文件必须有版本控制：

| 版本策略 | 说明 | 适用场景 |
|:---|:---|:---|
| **语义化版本** | MAJOR.MINOR.PATCH | 复杂资源，频繁变更 |
| **日期版本** | YYYY-MM-DD | 简单资源，低频变更 |
| **哈希版本** | Git commit hash | 开发中资源，频繁迭代 |

> 为什么版本控制：资源型文件会演进。不版本控制，使用者不知道使用哪个版本，可能导致填充后输出格式错误。

## 5. 资源型文件的内容要求

### 5.1 必须说明的内容

- **每个占位符必须有约束**：不定义约束，填充值无法验证。
- **每个占位符必须有必填/可选标记**：不标记，使用者不知道哪些必须填充。
- **模板必须有版本号**：不版本控制，演进时无法追踪。

### 5.2 禁止的内容

- **禁止无约束的占位符**：如 `{{content}}` 无类型、无长度约束。
- **禁止嵌套占位符**：如 `{{{{name}}}}`，增加解析复杂度。
- **禁止逻辑表达式**：资源型文件是静态模板，不应包含条件判断、循环等逻辑。

---

## 6. 与输出模板的协同

资源型文件与 output-template.md 的关系：

| 文件 | 职责 | 关系 |
|:---|:---|:---|
| **output-template.md** | 定义输出结构（简单/中等/复杂） | 源头：定义"需要什么资源" |
| **assets/{template}.json** | 提供具体模板内容 | 消费：被 output-template.md 引用 |
| **assets/{schema}.yaml** | 提供验证 schema | 验证：验证填充后输出合规 |

> 为什么分层：output-template.md 定义"需要什么"，assets/ 提供"具体是什么"。output-template.md 变更时，assets/ 同步更新。

---

## 7. 完整性检查清单

设计资源型文件时，逐项检查：

- [ ] 模板结构可验证（JSON Schema / XML Schema / YAML Lint）
- [ ] 所有占位符有定义（名称、类型、约束、必填）
- [ ] 占位符命名符合规范（双大括号、kebab-case、语义化、唯一性）
- [ ] 填充规则明确（必填/可选、验证方法、默认值）
- [ ] 模板有版本号（语义化版本/日期版本/哈希版本）
- [ ] 无嵌套占位符
- [ ] 无逻辑表达式
- [ ] 文件 < 200 行，超过则拆分
- [ ] 使用标准 Markdown 表格，无 ASCII 艺术线
