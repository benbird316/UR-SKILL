# 验证类 ref 设计指南

> 只教怎么写验证类 ref 文件。验证类 ref 回答"是不是真的？"——真假阳性模式 + 边界案例 + 验证方法论。
> 判定是否需要生成验证类 ref，见 skill-package-design-guide.md §2。

---

## §1 格式模板

```markdown
> 加载阶段：步骤 3

## {验证模式名称}

**正例**（真正的问题）：
- 特征：{可观测特征}
- 示例：{代码/场景片段}

**反例**（假阳性）：
- 容易误判为问题的正常模式
- 区分方法：{具体到哪个信号/指标区分}

**边界案例**：{需要人工判断的灰色地带 + 默认建议}
```

---

## §2 规则

- **MUST** 同时包含正例和反例——缺少任何一侧，Agent 无法校准
- **MUST** 区分方法具体到可观测信号，禁止"请人工判断"
- **MUST** 边界案例给默认建议（倾向于判真还是判假）
- **MUST NOT** 写修复方法（那是方案类的事）
- **SHOULD** ≤ 180 行。验证类通常最丰富。超过则检查是否混入了修复策略或方法论展开。

---

## §3 故障诊断文件（三底线范畴，非验证类 ref）

> 注意：故障诊断文件（troubleshooting.md）属于**三底线**（SKILL 运行质量保障），不属于领域知识的验证类 ref。二者范畴不同。

故障诊断文件记录 SKILL 执行过程中反复出现的失败模式及修复方法。

### 3.1 三个必要要素

| 要素 | 说明 | 为什么必须 |
|:---|:---|:---|
| **症状** | 用户/LLM 能观察到的具体表现 | 诊断的入口 |
| **根因** | 导致该症状的底层原因 | 只修症状不修根因→下次还坏 |
| **修复动作** | 具体、可执行的操作步骤 | 知道"为什么坏"但不知道"怎么修"= 无价值 |

### 3.2 格式模板

```markdown
### {编号} {故障名称}

- **症状**：{用户/LLM 能观察到的具体表现，用具体的错误信息、行为异常描述}
- **根因**：{导致症状的底层原因，解释"为什么会这样"}
- **修复动作**：
  1. {具体步骤 1}
  2. {具体步骤 2}
  3. {具体步骤 3}
```

### 3.3 按域分组

故障按**故障域**分组（与工作流步骤或能力域对应），便于 LLM 按上下文快速定位。

```
## 1. 能力矩阵相关故障
## 2. 工作流相关故障
## 3. 内容与格式相关故障
## 4. 质量与安全相关故障
```

### 3.4 编号系统

每个故障条目分配 **T{序号}** 编号（T = Troubleshooting），便于跨文件引用。
如：
```markdown
### T01 能力矩阵写成工作流步骤
```

### 3.5 与反模式的关系

| 维度 | 反模式（anti-patterns.md） | 故障诊断（troubleshooting.md） |
|:---|:---|:---|
| 阶段 | 设计时（预防） | 运行时（修复） |
| 触发 | 校验步骤的静态扫描 | 执行过程中的异常匹配 |
| 关注点 | "怎么做是错的" | "坏了怎么修" |
| 因果方向 | 做法 → 危害 | 症状 → 根因 → 修复 |

**映射关系**：一个反模式可能对应零个或多个故障条目。不是每个反模式都需要故障条目——只有那些必然导致可观察症状的反模式才需要。

---

## §4 检查清单

- [ ] 每个验证模式有正例和反例
- [ ] 区分方法具体到可观测信号
- [ ] 边界案例有默认建议
- [ ] 没有混入修复方法（修复归方案类）
- [ ] 文件 ≤ 180 行，超过则拆分
