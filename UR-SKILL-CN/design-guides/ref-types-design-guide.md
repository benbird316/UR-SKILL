# 参考文件设计指南

> 核心原则：一个 ref 文件 = 一个工作流步骤的知识支撑。Agent 按需加载，用后即释。
> 本指南只定义五类 ref 文件的格式规则。判定是否需要生成 ref 文件，见 skill-package-design-guide.md §2。

---

## §0 前置：5 类 ref 在文件体系中的位置

**5 类 ref ≠ 所有参考文件**。它们是「领域知识类参考文件」的细分，按工作流阶段和用途分类。

整个 SKILL 产物的参考文件体系分为：

| 层级 | 文件类型 | 说明 |
|:---|:---|:---|
| **三底线文件** | example.md / anti-patterns.md / troubleshooting.md | 每个完整 SKILL 基本都有（例外：单文件 SKILL、仅生成系统提示词、用户明确不需要） |
| **领域知识 5 类 ref** | 说明类 / 方法类 / 验证类 / 方案类 / 规范类 | 按需添加，针对具体领域知识，按工作流阶段一一对应 |
| **其他独立文件** | glossary.md 术语表 | 自创术语多或跨领域时添加 |

**容易混淆的区分：**
- 「验证类 ref」(verification-patterns.md) = 验证领域知识的真假，属于**领域知识**范畴
- 「故障诊断文件」(troubleshooting.md) = SKILL 运行时出问题怎么修，属于**三底线**范畴
- 「方案类 ref」(fix-patterns.md) = 正面的解决方案模式，属于**领域知识**范畴
- 「反模式文件」(anti-patterns.md) = 反面的错误模式，属于**三底线**范畴

## §1 五类 ref 文件

> 5 类 ref 的存放位置、触发条件、设计指南映射详见 [skill-package-design-guide.md §3](skill-package-design-guide.md) 文件类型速查表。本节仅补充工作流阶段和"回答什么"的语义说明。

| 类型 | 工作流阶段 | 回答 |
|:---|:---|:---|
| 说明类 | 分析/解析 | "这是什么？" |
| 方法类 | 扫描/识别 | "怎么找？" |
| 验证类 | 验证/校准 | "是不是真的？" |
| 方案类 | 修复/执行 | "怎么做？" |
| 规范类 | 输出/交付 | "怎么输出？" |

---

## §1.1 知识来源分类（K1-K4，供 research-analyst 判定）

research-analyst 调查知识源时，按内容类型分为四类。同一类知识源，因**用途**不同，落到不同的 ref 类型：

| K类 | 知识来源类型 | 典型示例 | 映射到哪种 ref（取决于用途） |
|:---|:---|:---|:---|
| **K1** 领域知识 | 概念体系、规则、方法论、最佳实践 | OWASP（安全）、PEP 8（Python）、小红书排版规范 | 说明类（"这是什么？"）**或** 方法类（"怎么找违反项？"） |
| **K2** 接口/工具参考 | API 签名、CLI 命令、函数参数 | React API、AWS CLI、PostgreSQL 语法 | 方法类——"怎么调用？参数是什么？" |
| **K3** 策略/配置 | 合规要求、安全策略、企业规范 | SOC 2 策略、GDPR 条款、内部编码规范 | 说明类——"规则/边界是什么？" |
| **K4** 设计模式/架构 | 可复用方案、范式、架构模式 | 微服务架构、状态机模式、MVC | 方案类——"怎么做？正反对照？" |

> **判定规则**：一个知识源 = 一个 K 类。同一个 K1 知识（如 OWASP SQL 注入），若目的是"让 LLM 理解概念"→ 说明类 ref；若目的是"教 LLM 定位代码中的 SQL 注入"→ 方法类 ref。K2 天然偏向方法类，K3 天然偏向说明类，K4 天然偏向方案类。K1 需要 research-analyst 明确用途后再分类。

## §1.2 参考文件设计三原则

所有 5 类 ref 文件共享以下设计原则：

| 原则 | 内容 | 不合格示例 |
|:---|:---|:---|
| **原子性** | 一个条目 = 一个单元主题，内部无交叉引用 | "详见 §2.3.1"（文件内自引用，读起来跳转） |
| **可引用性** | 每个条目有唯一的基于路径的标识符（如 `T01`、`AP-01`），workflow 可直接引用 | 无编号的纯标题，引用时只能说"大概在中间那部分" |
| **时效性** | 每个条目标注信源 + 获取日期；知识源不稳定的领域需定期刷新 | "根据最新研究……"（无日期，无法判断是否过期） |

> 原子性：Agent 一次只读一个条目，不应因看到一个引用再回头翻前文。
> 可引用性：SKILL.md 写 `见 ref §T03`，Agent 可精确跳转。
> 时效性：LLM 可通过日期判断知识是否可能过期。

## §2 步骤绑定

每个工作流步骤对应 0-1 个 ref 文件。步骤和 ref 类型一一绑定：

```
步骤 1（分析）→ 说明类 → classification.md
步骤 2（扫描）→ 方法类 → detection-methods.md
步骤 3（验证）→ 验证类 → verification-patterns.md
步骤 4（修复）→ 方案类 → fix-patterns.md
步骤 5（输出）→ 规范类 → output-spec.md
```

**MUST** 命名使用工作流阶段名，禁止知识主题命名（如 `rca-methodology.md`）。
**MUST** 相邻步骤无法清晰切分时合并，命名 `phase-{N}-{M}-{purpose}.md`，文件头标注加载步骤范围。

---

## §3 渐进式加载契约

Agent 执行模式：**一次一步，一文件一事**。

```
进入步骤 1 → 加载 classification.md → 完成 → 释放
进入步骤 2 → 加载 detection-methods.md → 完成 → 释放
...
```

**ref 文件头部契约声明**（必须）：
```markdown
> 加载阶段：步骤{N} | 进入步骤 N 时加载，完成后释放
```

**工作流步骤中引用**（必须）：
```markdown
**读取文件**：`references/{文件名}.md` — 用途：{一句话}
```

**MUST NOT** 笼统声明（"读取 references/ 下所有文件"）。
**MUST NOT** 步骤 1 加载步骤 4 的 ref。
**MUST NOT** ref 内容跨步骤（如 classification.md 中混入修复策略）。

---

## §4 长度约束

- 单个 ref **MUST** ≤ 200 行
- 超限 → 检查是否跨了步骤边界（主要根因）
- 确认未跨边界仍超限 → 步骤内按子主题拆：`classification-p1.md`
- **MUST NOT** `phase-01-all.md` 把所有步骤知识塞入一个文件

- [ ] 每个 ref 头部是否有加载阶段声明？
- [ ] 每个工作流步骤是否声明了 `**读取文件**` 且指定具体文件名？
- [ ] ref 内容是否严格对应单一步骤？
- [ ] 单个 ref 是否 ≤ 200 行？
