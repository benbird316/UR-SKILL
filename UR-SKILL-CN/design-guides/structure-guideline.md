# 结构规范指南

> 用途：定义 SKILL.md 及配套文件的格式符号、信息组织方式、注意力对抗策略
> 核心原则：规范是"做法"不是"目标"；遵循 RFC 2119；对齐业界 Markdown 标准

---

## 1. 通用格式符号

本文件使用的格式符号均来自 Markdown 标准或 RFC 2119，不引入自创符号。

### 1.1 约束等级符号（RFC 2119）

| 符号 | 含义 | 来源 | 偏离处理 |
|:---|:---|:---|:---|
| **MUST** | 绝对要求 | RFC 2119 | 违反 = 不兼容规范 |
| **MUST NOT** | 绝对禁止 | RFC 2119 | 违反 = 不兼容规范 |
| **SHOULD** | 强烈建议 | RFC 2119 | 偏离需记录理由 |
| **SHOULD NOT** | 不推荐 | RFC 2119 | 偏离需记录理由 |
| **MAY** | 真正可选 | RFC 2119 | 无偏离概念 |

> 核心原理见 [design-rationale/design-rationale.md §6](../design-rationale/design-rationale.md#L75)。简言之：RFC 2119 关键词全大写时具有精确语义，可被工具机械解析。

### 1.2 检查符号

| 符号 | 含义 | 使用场景 |
|:---|:---|:---|
| `- [ ]` | 未检查 | 检查清单、审视项 |
| `- [x]` | 已检查 | 检查清单完成标记 |
| `1.` `2.` `3.` | 步骤编号 | 工作流序列 |
| `规则01` `规则02` | 规则编号 | 规则引用 |
| `反模式1` `反模式2` | 反模式编号 | 反模式引用 |
| `门控1` `门控2` | 门控编号 | 门控检查点引用 |
| `风险边界-01` | 风险边界编号 | 风险边界声明引用 |

> 为什么用编号系统：便于跨章节引用，门控检查项引用规则编号而非重复内容。

### 1.3 强调符号

| 符号 | 含义 | 使用场景 |
|:---|:---|:---|
| `**加粗**` | 关键约束 | 规则关键词、检查项核心词 |
| `> 引用块` | 核心原则 | 需要强调的原则性内容 |
| `` `行内代码` `` | 格式示例 | 文件路径、配置项 |
| ` ``` ` 代码块 | 完整示例 | YAML frontmatter、配置模板 |
| `---` | 模块分隔 | YAML frontmatter 分隔、章节分隔 |

### 1.4 不使用的符号

以下符号不用于本体系，避免增加认知负担和编码风险：

- **不用 emoji（🔴 🟡 🟢）作为约束标记**：无法被所有终端解析，无统一语义
- **不用 Unicode 特殊符号（如 ◈）**：编码风险，无业界对应，无法被工具解析
- **不用 ASCII 艺术表格线（├ ┌ ┬）**：Markdown 标准表格 `|` 足够，ASCII 艺术增加噪音

## 2. 信息组织方式

### 2.1 一维信息用列表

**什么是一维信息**：步骤序列、规则声明、检查清单、单一属性列表。

**怎么做**：
- 规则声明：bullet list，每条独立，以 RFC 2119 关键词开头
- 步骤序列：编号列表 `1. → 2. → 3.`
- 检查清单：`- [ ]` 复选框

**为什么**：表格增加格式噪音（分隔线、对齐线等占显著比例），列表信息密度更高，且是 Markdown 标准。

### 2.2 二维信息用表格

**什么是二维信息**：两个维度的映射关系，如对比、门控映射、预算分配。

**怎么做**：用标准 Markdown 表格 `| col1 | col2 | col3 |`。

**为什么**：二维关系用表格一目了然，但仅限真正二维的信息。一维信息强制表格化会降低密度。

**以身作则**：本文件 1.1 节的约束等级是二维（符号×含义），用表格；2.1 节的列表做法是说明性文字，用 bullet list。

### 2.3 层次递进

**怎么做**：
- L1 总纲：段首 1-2 句，说明文件用途和核心原则
- L2 模块：`##` 二级标题，划分独立主题
- L3 条目：列表或表格，具体规范条目
- L4 细节：移入 `references/`，不在 body 中展开

**为什么**：渐进式加载，body 只保留骨架，细节按需加载，控制 token 消耗。

## 3. 正向表述

### 3.1 硬约束

**怎么做**：以 **MUST** / **MUST NOT** 开头，具体可执行，避免抽象。

**反例 → 正例**：

| 反例（反向/抽象） | 正例（正向/具体） |
|:---|:---|
| 不得丢失原文语义 | **MUST** 保留原文语义 |
| 禁止生成可执行代码 | **MUST NOT** 在 body 中嵌入可执行代码 |
| 不能把示例直接填充 | **MUST NOT** 将示例直接复制到 body |
| 不得出现内部术语 | **MUST NOT** 在 body 中使用内部术语 |

### 3.2 强偏好

**怎么做**：以 **SHOULD** / **SHOULD NOT** 开头，有例外时记录理由。

**反例 → 正例**：

| 反例 | 正例 |
|:---|:---|
| 避免格式冲突 | **SHOULD** 确保格式一致性 |
| 不要跳过任何步骤 | **SHOULD** 按顺序执行每个步骤 |
| 避免信息冗余 | **SHOULD** 控制信息密度，删除模式化填充 |
| 不要过度推断 | **SHOULD NOT** 超出用户输入进行推断 |

### 3.3 可选

**怎么做**：以 **MAY** 开头，真正可选，无义务亦无影响。

## 4. 注意力对抗策略

### 4.1 位置策略（对抗 Lost in the Middle）

> 核心原理见 [design-rationale/design-rationale.md §5](../design-rationale/design-rationale.md#L64)

**怎么做**：
- 文件首段：放核心能力摘要 + 关键约束
- 文件尾段：放检查清单 + 关键限制 + 版本信息
- 避免：关键规则放在 body 中间段

### 4.2 重复策略（对抗 Attention Dilution）

**怎么做**：
- 核心规则在规则章节声明，在工作流中引用编号
- 关键输出格式在规范中定义，在示例中展示
- 不同表述，避免 verbatim 重复

**为什么**：关键信息在不同位置以不同方式出现，增加被捕获概率，同时避免 verbatim 重复导致的注意力疲劳。

### 4.3 分段策略（对抗 Context Rot）

> 核心原理见 [design-rationale/design-rationale.md §5](../design-rationale/design-rationale.md#L64)

**怎么做**：
- body 超过 500 行 → 拆分内容至 `references/`
- `references/` 文件超过 200 行 → 拆分为多个小文件
- 单个表格行数过多 → 拆分为多个表格或分组
- 单个列表项数过多 → 分组或拆分为子列表

> 500 行 / 200 行对应 design-rationale/design-rationale.md 中 L2 / L3 的 token 预算阈值。

### 4.4 信号强化

**怎么做**：
- 关键约束用 `**加粗**`
- 核心原则用 `> 引用块`
- 格式示例用 ` ``` ` 代码块
- 规则 / 门控 / 反模式用编号系统（规则01, 门控1, 反模式1）

**为什么**：标准化标记降低认知负担，提高可解析性。加粗和引用块是 Markdown 标准，任何平台均可渲染。

### 4.5 Body 区块推荐排列顺序（双重锚定策略）

利用 LLM 的 U 型注意力曲线（首因效应 + 近因效应），将 body 内容按注意力区块分配：

| 区块 | 位置 | 放什么 | 原因 |
|:---|:---|:---|:---|
| **首因区（Primacy Zone）** | body 前 20% | 角色声明、任务定义、核心约束（MUST/MUST NOT） | 注意力最高，决定模型的行为基调 |
| **中间区（Middle Zone）** | body 中间 60% | 能力矩阵、能力切面、工作流步骤 | Lost in the Middle 风险区，放索引和引用路径 |
| **近因区（Recency Zone）** | body 后 20% | 输出格式契约、关键规则摘要（Double Prompting）、风险边界声明、参考引用路径 | 次高注意力，核心规则在末尾重复增加捕获概率 |

**Double Prompting 策略**（零成本增强）：
- 在近因区**用不同表述重复**首因区的核心约束（不写 verbatim 副本）
- 示例：首因区写 "MUST NOT 将能力矩阵与工作流步骤混淆"，近因区写 "规则 规则15：禁止架构混淆（反模式4）"

> 核心原理见 [design-rationale/design-rationale.md §5](../design-rationale/design-rationale.md#L64)

**反例**：
- 错误：规则（MUST/MUST NOT）放在 body 中部 → 最容易丢失的区域
- 错误：能力矩阵细节放在首因区 → 消耗了宝贵的注意力资源
- 错误：结尾只有"参考引用"没有核心约束的重复 → 浪费了近因区

> 参考：Liu et al., "Lost in the Middle" (ACL 2024); Google Research, "Double Prompting" (2025)

## 5. 信息密度

### 5.1 原则

保持高信息密度：删除模式化填充、避免冗余声明、每个句子传递独立信息。

### 5.2 检测方法

- 删除"众所周知""需要注意的是"等模式化前缀
- 删除重复声明（如规则在工作流中重复写出内容，应引用编号）
- 删除无信息增量的过渡句
- 检查是否每个段落都有独立信息价值

### 5.3 为什么不用数值阈值

信息密度是设计原则，不是可量化指标。文本领域没有 universally accepted 的计算公式（熵值法、压缩比、有效词比例均受格式和语料影响）。定性描述比拍脑袋数值更可靠。

## 6. 结构检查清单

生成 SKILL 后，逐项检查：

- [ ] 规则使用 RFC 2119 关键词（MUST / MUST NOT / SHOULD / SHOULD NOT / MAY）
- [ ] 一维信息用列表，二维信息用表格
- [ ] 层次递进：L1 总纲 → L2 模块 → L3 条目 → L4 细节
- [ ] 无自创符号（emoji、Unicode 特殊符号、ASCII 艺术表格线）
- [ ] 正向表述：MUST / SHOULD / MAY 替代 "不得/禁止/必须"
- [ ] 关键信息在首尾段重复或引用
- [ ] 无抽象规则（如"做好工作"）
- [ ] 无矛盾规则（如 规则01 说 MUST A，规则02 说 MUST NOT A）
- [ ] 规则、门控、反模式、风险边界四者职责分离，不重叠
- [ ] 门控检查项引用规则编号，不重复规则内容
