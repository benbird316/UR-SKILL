# 脚本文件设计指南

> 用途：定义 scripts/ 目录下可执行脚本的安全规范、设计方法和内容要求
> 核心原则：LLM 生成并可能自动执行的代码，安全是底线

---

## 1. 为什么需要脚本文件设计指南

scripts/ 目录下的脚本由 LLM 生成，可能由自动化系统执行。与人工编写的代码不同：
- LLM 可能生成不安全的代码（文件删除、网络请求、系统命令）
- 自动化系统可能无确认直接执行
- 执行环境可能是生产环境

如果不定义安全规范：
- 脚本可能删除用户数据
- 脚本可能泄露敏感信息
- 脚本可能执行恶意操作

脚本文件是**工具型参考文件**（L3），按需加载。当 SKILL 需要执行自动化操作时，加载脚本文件。

---

## 2. 脚本的安全规范

### 2.1 五个安全原则

| 原则 | 说明 | 为什么必须 |
|:---|:---|:---|
| **只读优先** | 脚本优先读取，不修改文件系统 | 修改操作风险最高，只读操作即使出错也可恢复 |
| **幂等性** | 多次执行结果一致 | 自动化系统可能重复执行，非幂等脚本导致重复修改 |
| **自包含** | 不依赖外部环境 | 执行环境不确定，外部依赖可能导致脚本失败或执行意外操作 |
| **最小权限** | 只请求必要的权限 | 权限越大，风险越大；最小权限原则限制潜在危害范围 |
| **输入验证** | 验证所有输入，拒绝非法输入 | LLM 生成的输入可能包含恶意内容，验证防止注入攻击 |

> 为什么五个原则：缺少任何一个，脚本安全性无法保证。只读优先是最低安全底线；幂等性确保可重复执行；自包含确保环境无关；最小权限限制危害范围；输入验证防止注入。

### 2.2 禁止的操作

- **禁止文件系统修改**：删除、写入、重命名文件/目录
- **禁止网络请求**：HTTP/HTTPS 请求、API 调用、下载
- **禁止系统命令执行**：shell 命令、子进程、系统调用
- **禁止环境变量修改**：修改 PATH、修改系统配置
- **禁止敏感信息访问**：读取密码、密钥、Token、配置文件

> 为什么禁止这些操作：这些是"高危操作"。LLM 生成的脚本无法保证这些操作的安全性，必须禁止。

### 2.3 允许的操作

- **读取文件**：只读访问，不修改
- **正则匹配**：文本扫描、模式识别
- **字符串处理**：分割、拼接、格式化
- **数学计算**：数值运算、统计
- **数据结构操作**：列表、字典、集合操作

> 为什么允许这些操作：这些操作是"只读"或"纯计算"，不产生副作用。即使脚本出错，也不会破坏数据或泄露信息。

## 3. 生成方法（ur-skill特殊说明）

> 本 SKILL 是"设计 SKILL 的 SKILL"（ur-skill），自身没有历史项目积累。
> 因此，脚本不是"从历史项目收集"，而是"基于生成的 SKILL 的核心场景，动态设计"。

### 3.1 基于场景需求生成

当使用本 SKILL 生成具体 SKILL 时，按以下步骤生成脚本：

1. **确定脚本需求**：从用户需求中提取是否需要自动化检测/验证/生成
   - 简单 SKILL：通常不需要脚本（无自动化需求）
   - 中等 SKILL：可能需要 1-2 个验证脚本
   - 复杂 SKILL：可能需要 3-5 个脚本（扫描/验证/生成/转换）

2. **设计脚本类型**：
   - 扫描脚本：检测反模式、占位符
   - 验证脚本：验证格式、约束、合规性
   - 生成脚本：批量生成、自动填充（中风险，需人工审核）
   - 转换脚本：格式转换、数据迁移（中风险，需人工审核）

3. **遵循安全规范**：
   - [ ] 脚本安全等级标注（安全/中风险/高风险）
   - [ ] 脚本只读优先，不执行高危操作
   - [ ] 脚本输出到 stdout，不直接写入文件
   - [ ] 脚本有输入验证和错误处理

### 3.2 基于用户指示生成

如果用户明确要求脚本功能：

1. **分析用户需求**：明确脚本类型、输入、输出、执行环境
2. **设计脚本结构**：按模板设计脚本
3. **安全审查**：
   - [ ] 脚本不执行文件删除/写入/网络请求/系统命令
   - [ ] 脚本有 SAFE_MODE 开关（默认 True）
   - [ ] 中风险及以上脚本需人工审核

### 3.3 ur-skill 验证脚本

ur-skill 设计 SKILL 时，使用以下脚本验证生成质量（详细执行流程见 ur-skill.md 工作流校验步骤）：

| 脚本 | 用途 | 安全等级 | 执行时机 |
|:---|:---|:---:|:---|
| 反模式扫描 | 检测生成的 SKILL 是否违反反模式 | 安全 | 交付前 |
| 占位符检测 | 检测 body 中是否有未填充占位符 | 安全 | 交付前 |
| 审视检查计数 | 检测工作流步骤中审视检查数量是否匹配节点类型 | 安全 | 交付前 |
| 信息密度评估 | 评估 body 信息密度是否高 | 安全 | 交付前 |

> 为什么ur-skill需要脚本：ur-skill设计 SKILL 时，需要自动化验证生成质量。这些脚本是ur-skill的"自检工具"，确保生成的 SKILL 符合规范。

## 4. 脚本文件的设计方法

### 4.1 脚本类型

| 类型 | 用途 | 安全等级 | 示例 |
|:---|:---|:---:|:---|
| **扫描脚本** | 静态检查，识别问题 | 安全 | 反模式扫描、占位符检测 |
| **验证脚本** | 验证格式、约束、合规性 | 安全 | YAML frontmatter 验证、表格格式检查 |
| **生成脚本** | 生成内容、填充模板 | 中风险 | 批量生成 SKILL 文件、自动填充占位符 |
| **转换脚本** | 格式转换、数据迁移 | 中风险 | Markdown 转 YAML、JSON 转 CSV |
| **部署脚本** | 部署到生产环境 | 高风险 | 自动部署、CI/CD 流水线 |

> 为什么分安全等级：扫描和验证脚本是只读操作，安全。生成和转换脚本可能修改文件，中风险。部署脚本直接操作生产环境，高风险。高风险脚本需要人工审核，不能自动执行。

## 5. 脚本与反模式的协同

脚本用于自动化检测反模式：

| 反模式 | 检测脚本 | 执行时机 |
|:---|:---|:---:|
| 反模式2 占位符残留 | 正则扫描脚本 | 交付前 |
| 反模式6 检查缺失 | 审视检查计数脚本 | 交付前 |
| 反模式2 规范越位 | 表格结构扫描脚本 | 交付前 |

> 为什么交付前执行：脚本检测是静态检查，与门控（动态检查）互补。脚本可集成到 CI/CD 流程中，自动化拦截问题。
